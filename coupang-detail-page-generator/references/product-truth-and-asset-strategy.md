# 제품 사실·정체성·자산 전략

## 제품 원본 역할

| 역할 | 기존 별칭 | 사용 | 금지 |
|---|---|---|---|
| `PRODUCT_SOURCE` | `PRODUCT_TRUTH`, `PRODUCT_ANGLE` | 형태·색·부품·라벨·각도 기준 | 배경·인물 정체성 참조 |
| `SUPPLIER_REFERENCE` | 없음 | 공급처 사실·연출 참고 | 신뢰·SKU 확인 없이 외형 기준으로 승격 |
| `COMPETITOR_REFERENCE` | `COMPETITOR` | 시장·증거 모듈 조사 | ImageGen 제품 입력·최종 합성 |
| `STYLE_REFERENCE` | `REFERENCE_LAYOUT` | 구성·조명·밀도·타이포 원리 | 상품 사실·브랜드·카피 이전 |
| `GENERATED_DRAFT` | 없음 | 실패 영역 편집 대상 | 제품 사실·동일성 근거 |
| `MEASUREMENT_EVIDENCE` | 없음 | 수치·표시사항 근거 | 적용 모델·조건을 넘는 일반화 |
| `REAL_PHOTO_TARGET` | 없음 | 후속 촬영으로 교체할 위치 | 임시 AI 이미지를 최종 증거로 주장 |

## 제품 정체성 잠금

다음을 관찰 가능한 문장으로 쓴다.

- 전체 실루엣과 앞·뒤·좌·우 비율
- 정확한 색상과 투명도
- 보이는 소재 결·두께 인상
- 주요 부품 수와 상대 위치
- 비대칭 요소와 결합 방향
- 밑창·표면 패턴
- 주름·라벨·봉제·밴딩·버클·단추
- 사용 시 변형되는 부분과 고정되는 부분
- 생성 모델이 자주 틀리는 요소
- 절대 생성하면 안 되는 유사 소재·유사 모델·추가 부품

## Canonical reference와 장별 선택

제품마다 실제 소스 3~5장을 canonical reference로 고르고 `reference-routing.json`에 등록한다. 정면·후면·좌우·상하단·매크로·열림/닫힘 중 실제 페이지가 필요한 면을 우선한다. 각 페이지는 canonical 안에서 보존할 면을 가장 잘 보여주는 1~5개 source ID를 선택하며, 선택 이유와 부족한 면을 함께 기록한다. 같은 묶음을 10장에 반복하거나 생성 이미지를 canonical source로 넣으면 실패다.

## 제품 소스 계보

모든 소스와 생성 자산을 `product-source-lineage.json`에 기록한다.

- `raw_capture`: 사용자가 직접 촬영한 실제 SKU
- `manufacturer_source`: 제조사·공급처가 제공한 실제 SKU 사진
- `generated_master`: 실제 자료를 바탕으로 AI가 다시 생성한 제품
- `generated_scene`: 제품을 포함해 AI로 만든 연출 장면

제품 동일성은 `raw_capture` 또는 출처·모델이 명확한 `manufacturer_source`로만 확정한다. `generated_master`와 `generated_scene`은 제품 구조의 근거가 아니다. 사람 없는 제품 입력이라는 이유만으로 실제 SKU 근거로 승격하지 않는다.

소스별로 `source_id`, `lineage_type`, `path_or_url`, `sku_or_model`, `trusted_for_identity`, `contains_person`, `captured_at`, `notes`를 기록한다. 생성 자산은 `asset_id`, `source_ids`, `generation_stage`, `concept_only`, `identity_check`를 기록한다.

## 동일성 검증 게이트

생성·합성 전에 다음을 실제 제품 자료와 대조한다.

- 전체 실루엣과 비율
- 앞코와 발목·입구 형태
- 스트랩 개수·위치·폭·길이
- 스냅·버클·잠금장치 구조
- 절개선·봉제선·몰딩선
- 밑창 형태·무늬·돌기
- 각인·라벨·로고 위치
- 색상·투명도·재질 표현
- 구성품 개수와 결합 방식

하나라도 명확하지 않거나, 생성 결과가 실제 소스와 다르거나, 단계별 이미지에서 구조가 달라지면 `BLOCKED`다. 제품 이미지를 임의로 생성하지 말고 확인이 필요한 부위만 촬영 요청한다.

요청 문구는 다음을 기본으로 한다.

> 현재 자료만으로는 실제 제품의 `[확인 불가능한 부분]`을 정확히 확인할 수 없습니다. 제품 디테일이 중요한 작업이라 AI로 임의 추정하지 않겠습니다. `[필요한 촬영 각도와 부위]`를 촬영해 주시면 실제 제품 구조를 고정한 뒤 작업을 계속하겠습니다.

기본 촬영은 정면, 좌우 측면, 앞코·입구, 잠금 전체 열림·닫힘, 밑창 전체, 각인·라벨·로고 확대, 전체 구성품, 실제 착용·사용 순서다. 밝은 곳의 무보정 원본으로 제품이 잘리지 않게 받고 작은 구조물은 근접 컷을 추가한다.

사용자가 촬영 없이 진행을 명시하면 `concept_only_opt_in: true`로 기록한다. 결과에는 `연출용 콘셉트 이미지`를 표시하고 동일 제품의 실제 구조·사용법·상세 증거로 사용하지 않는다.

프롬프트에는 `절대 변경하지 않을 항목`으로 반복한다. 생성 후 부품 수·위치, 좌우 모델, 색·투명도, 앞코·뒤꿈치·높이 비율, 밑창, 임의 로고·지퍼, 제품과 소품의 융합을 확인한다.

## 자산 모드

| 모드 | 사용 조건 | 최종 증거 가능 여부 |
|---|---|---|
| `PRESERVE` | 원본 제품 픽셀을 그대로 사용 | 가능 |
| `COMPOSITE` | 사람 없는 제품을 분리해 새 배경에 합성 | 가능 |
| `EDIT_BACKGROUND_ONLY` | 제품을 고정하고 배경·조명만 변경 | 가능 |
| `GENERATE_SUPPORT` | 배경·소품·아이콘·일반 상황 | 보조만 가능 |
| `GENERATE_PRODUCT_ALLOWED` | 원본과 달라도 오인이 없는 낮은 위험 표현 | 제한적 |
| `REAL_PHOTO_REQUIRED` | 실측·포장·마감·동작·성능 증명 | 촬영 후 가능 |
| `GIF_REQUIRED` | 정지 이미지보다 동작 증거가 핵심 | 실사 모션 우선 |

## 강제 판정

다음은 `PRESERVE` 또는 `COMPOSITE`를 우선한다.

- 제품이 화면 25% 이상인 히어로
- 펼침·결합·잠금·조절 구조
- 밑창·라벨·옵션·사이즈 비교
- 첫 장과 마지막 구매 결정 장
- 원본과 다르면 고객이 다른 제품으로 오인하는 장면

다음은 `REAL_PHOTO_REQUIRED`를 우선한다.

- 사이즈별 실측, 두께, 무게
- 실제 구성품·포장
- 실제 착용 순서
- 세척·건조·보관 방법
- 밑창 전체와 마감 근접
- 시험 수치나 성능을 증명하는 장면

## 장별 자산 전략 표

| 장 | 요소 | 모드 | 원본 파일 | 허용 변경 | 금지 변경 | 왜곡 위험 | 임시 사용 | 최종 교체 컷 |
|---|---|---|---|---|---|---|---|---|

모든 주요 제품 요소와 증거 모듈을 한 행씩 기록한다. 동일 페이지 안에서도 제품은 `COMPOSITE`, 빗물은 `GENERATE_SUPPORT`, 실측은 `REAL_PHOTO_REQUIRED`처럼 분리할 수 있다.

## 사실-주장-증거 계약

최종 주장마다 다음을 연결한다.

`claim_id → 승인 카피 → 사실 상태 → 출처 → 페이지 → proof_type → asset_mode → 촬영/GIF 교체 여부`

연결이 끊긴 주장은 삭제한다. 그래픽 화살표·물방울 아이콘·분위기 장면만으로 기능을 입증하지 않는다.

## 최종 동일성 승인

- 실제 SKU와 제품 구조가 일치한다.
- 스트랩·잠금장치·절개선·밑창이 실제 소스와 일치한다.
- 여러 장면과 사용 단계에서 구조가 일관된다.
- 확인되지 않은 부품·기능·각인·로고가 추가되지 않았다.
- 모든 제품 자산에서 실제 소스부터 생성 결과까지 계보가 추적된다.
