# Browser Use 쿠팡 경쟁사 조사와 기획 원리

## 목적

Browser Use CLI로 실제 상품 페이지와 쿠팡 경쟁사 상세페이지를 직접 확인하고, 경쟁사 문구를 복사하지 않은 채 카테고리 구매 원리와 우리 제품의 검증 가능한 소구점으로 변환한다.

## 조사 순서

1. 작업 전 탭 목록과 이번 작업의 탭 식별자를 기록한다.
2. 상품 출처 URL을 먼저 열고 화면 캡처와 DOM 텍스트를 수집한다.
3. 제품명·모델·옵션·구성·소재·치수·상세 이미지 URL을 `source-product.json`에 기록한다.
4. 제품의 일반명, 구조명, 사용 상황을 조합해 쿠팡 검색어를 3개 이상 만든다.
5. 동일 제품·동일 모델, 같은 구조의 직접 경쟁상품, 다른 방식의 대체상품을 구분해 조사한다. 쿠팡 직접 경쟁사는 5개 이상 고르고 동일 OEM·동일 판매자만 반복 선택하지 않는다.
6. 각 상품 페이지를 열어 상단 훅, 상세 전개, 소구점, 시각 증거를 기록하고 상세 이미지 또는 스크린샷을 저장한다.
7. 접근 가능한 리뷰·문의에서 반복 긍정, 불만, 혼동, 반품 위험을 요약하고 빈도를 `높음/중간/낮음/확인 부족`으로 기록한다.
   - 상품별 실제로 연 리뷰 수, 접근 구간, 정렬 방식, 확인 시각을 `review_scope`에 기록한다.
   - 검색 결과의 짧은 문구만 확인했으면 `검색 스니펫 기반`으로 표시하고 전체 리뷰 분석처럼 쓰지 않는다.
8. 반복 주장, 과포화 주장, 구매 저항, 정보 빈틈을 교차 비교한다.
9. 우리 제품 근거와 교차검증해 승인·보류·금지 소구점으로 분류한다.
10. 조사 종료 시 이번 작업이 연 탭만 닫고 기존 사용자 탭을 보존한다.

## Browser Use 실행 규칙

- 첫 이동은 `new_tab(url)`을 사용하고 `wait_for_load()` 뒤 `capture_screenshot()`으로 화면을 먼저 본다.
- 좌표 클릭은 캡처 뒤에만 사용한다. 구조화된 정보 수집은 `js(...)`로 DOM을 읽는다.
- 로그인·비밀번호·MFA가 필요하면 멈추고 사용자에게 요청한다.
- 쿠팡 접근 차단이나 CAPTCHA가 나오면 우회하지 않고 상태를 기록한다.
- 조사 URL, 확인 시각, 검색어, 로컬 캡처 경로를 `sources.md`에 남긴다.
- 경쟁사 상세페이지 이미지는 조사 전용이며 ImageGen 입력과 최종 합성에서 제외한다.
- 작업 시작 전 탭과 새로 연 탭을 구분한다. 종료 시 새 탭만 닫으며 사용자 탭을 일괄 종료하지 않는다.

## 경쟁사 선정 기준

- 동일 카테고리와 사용 목적이 같은 직접 경쟁사일 것
- 가격만 유사한 다른 구조의 제품을 직접 경쟁사로 섞지 않을 것
- 상위 노출, 리뷰·판매 신호, 시각적 완성도, 구조 차이를 고르게 포함할 것
- 동일 상품의 판매자만 다른 중복은 한 개로 취급할 것

## 소구점 추출

각 경쟁사에서 다음을 분리한다.

- `observed_appeals`: 화면에 실제로 보이는 구매 이유
- `evidence_modules`: 확대, 실측, 전후 비교, 사용 단계, 옵션표처럼 주장을 뒷받침하는 표현
- `buyer_tension`: 젖음, 미끄러움, 휴대, 착용 번거로움 같은 구매 전 고민
- `market_gap`: 반복되지만 증거가 약한 주장 또는 경쟁사가 설명하지 않은 기준
- `review_positive`: 반복 긍정 요인과 근거 범위
- `review_pain`: 반복 불만·혼동·반품 위험과 빈도
- `review_scope`: 접근한 상품 수·리뷰 수·페이지 범위·스니펫 여부·확인 시각

리뷰 문장을 길게 복사하지 않는다. 한두 리뷰를 시장 전체로 일반화하지 않고 여러 상품·리뷰에서 반복되는지 확인한다. 리뷰 접근이 차단되면 `확인 부족`으로 남기며 검색 결과 요약만으로 리뷰 사실을 확정하지 않는다.

## 기획 원리 변환

소구점은 아래 사슬을 모두 충족할 때만 `approved`로 둔다.

`경쟁사에서 관찰됨 → 카테고리 구매 판단과 관련됨 → 우리 제품 사실로 입증됨 → 이미지로 증명 가능함`

경쟁사에서 자주 보여도 우리 제품 근거가 없으면 `rejected`와 `forbidden_transfers`에 기록한다. 브랜드 고유 문장, 인증, 후기, 판매량, 방수 등급, 미끄럼 방지, 내구 수치, 소재 성능은 우리 근거가 없으면 가져오지 않는다.

`planning-principles.md`에는 다음을 작성한다.

1. 구매자가 먼저 확인하는 기준 3~5개
2. 경쟁사 과포화 소구와 차별화 가능한 빈틈
3. 우리 제품 승인 소구점 3~5개와 각 증거
4. 첫 화면 훅과 10장 전환 흐름의 원리
5. 사용 금지 주장과 표현

여기서 “원리”는 소구점 목록이 아니라 페이지의 의사결정 규칙이다. 아래 연결이 한 문장과 구조표에서 모두 보여야 한다.

`시장 문제 → 구매 판단 기준 → 우리 제품의 답 → 증거 방식 → 페이지 전개`

`appeal-map.json`의 `planning_rule`에는 다음 필드를 반드시 둔다.

- `market_problem`: 고객이 겪는 손실·불편·불안
- `category_decision`: 구매 직전의 핵심 확인 질문
- `own_product_answer`: 검증된 우리 제품 사실로 만든 답
- `proof_strategy`: 사진·실측·구조·사용 순서 중 필요한 증거
- `page_flow`: 문제 인식부터 구매 확신까지의 페이지 순서

`planning-principles.md`에는 최소한 `## 한 줄 기획 원리`, `## 구매 판단 흐름`, `## 승인 소구`, `## 금지 전이` 제목을 둔다. 경쟁사에서 많이 보인 문구라도 이 연결에서 우리 제품 근거가 끊기면 승인하지 않는다.

## 완료 조건

- 쿠팡 직접 경쟁사 5개 이상
- 경쟁사별 URL·소구점·증거 모듈·상세페이지 로컬 자료 존재
- 승인 소구점마다 경쟁사 관찰 근거와 우리 제품 근거 존재
- 선택 소구점이 `planning-principles.md`와 10장 기획에 추적 가능
- 금지 소구점이 카피와 이미지 프롬프트에 나타나지 않음
- `competitor-pain-map.md`의 각 항목에 근거·반복 수준·우리 제품의 실제 응답 가능성·필요 증거가 있음
- 이번 작업이 연 Browser Use 탭이 닫히고 기존 사용자 탭이 보존됨
