# 쿠팡 커머스 오케스트레이터

전체 작업을 직접 수행하지 말고 프로젝트 정본을 읽어 현재 단계와 다음 게이트를 결정한 뒤 필요한 전문 스킬 하나만 호출한다.

## 시작 절차

1. `STATUS.md`, `docs/ROADMAP.md`, 대상 프로젝트 `output/README.md`와 manifest를 읽는다.
2. 가장 최신인 실제 프로젝트 파일의 상태를 우선하고, 문서 간 상태가 다르면 충돌을 보고한다.
3. 아래 상태 카드로 현재 위치를 먼저 보여준다.
4. 차단이 없으면 다음 전문 스킬로 진행한다. 사용자 결정이 필요하면 한 번에 하나의 질문만 한다.

```text
현재 단계: <n/10 · 단계명>
완료 근거: <파일 또는 QA>
차단 조건: <없음 또는 1개>
다음 AC: <통과해야 할 수치·승인>
다음 작업: <호출할 스킬과 산출물>
사용자 질문: <필요할 때만 한 문장>
```

## 라우팅

| 단계 | 전문 스킬 | 완료 신호 |
|---|---|---|
| 1 소싱·가격 승인 | `$coupang-product-sourcing` | 승인 후보·가격안·근거 해시 |
| 2~4 사실·시장·제품기획 | `$coupang-product-planning` | 사용자 승인된 `product-plan.json` |
| 5~9 콘텐츠기획·스토리보드·제작·HTML | `$coupang-content-studio` | 승인 계획, 소재 QA, 편집 가능 HTML |
| 10 게시 전 검수 | `$coupang-publish-qa` | 현재 패키지 해시에 묶인 자동·육안 QA |
| 호환 실행 | `$coupang-detail-page-generator` | 기존 5.3 프로젝트·공용 스크립트 |

전문 스킬을 여러 개 동시에 시작하지 않는다. 앞 단계 AC가 실패했으면 가장 가까운 원인 단계로만 되돌린다.

## 사용자 UX

- 한 번에 하나의 질문만 한다. 질문 없이 안전하게 진행할 수 있으면 질문하지 않는다.
- “모두 승인할까요?”처럼 서로 다른 결정을 묶지 않는다.
- 사용자에게 파일명을 암기시키지 말고 결정의 의미와 영향부터 설명한다.
- 승인 질문에는 대상 파일, 현재 해시, 승인 뒤 열리는 단계, 변경 시 승인이 무효화됨을 짧게 알린다.
- 진행 보고마다 `현재 단계`, `완료 근거`, `다음 AC`를 유지한다.
- 비용·발주·등록·광고 집행은 별도 명시적 승인 없이는 수행하지 않는다.

## Acceptance Criteria

- 현재 단계 판정이 manifest와 QA 근거에 연결됨: 100%
- 오래된 승인·QA를 다음 단계 통과로 사용한 건수: 0
- 동시에 제시하는 사용자 질문 수: 최대 1
- 미확정 사실을 판매 주장으로 승격한 건수: 0
- 다음 전문 스킬과 필요한 산출물이 상태 카드에 명시됨: 100%

세부 단계·상태 전이는 [references/stage-contract.md](references/stage-contract.md)를 읽고 따른다.
