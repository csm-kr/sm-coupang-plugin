# 쿠팡 상품 소싱 분석

도매 후보를 공개 근거로 조사하고 계산 가능한 값만 사용해 기준 통과 상품 5개 이상의 `QUALIFIED_POOL`을 만든다. 도매꾹 URL과 쿠팡 근거 URL이 포함된 보고서를 사용자에게 제시하고, 사용자가 진입할 상품을 선택한 뒤에만 최대 5개의 `SHORTLIST`로 승격한다. 발주, 결제, 공급사 연락, 쿠팡 등록은 수행하지 않는다.

워크플로 UI의 사용자 시작 모드는 `coupang-best-high-markup-sourcing` 하나로 고정한다. 이 스킬은 그 탐색 일치 후보를 전체 비용·수요·경쟁·운영 게이트로 재검증하는 내부 정본이다.

## 필수 원칙

- 수요를 가장 먼저 검증하고 마진, 경쟁 속 진입기회, 운영위험 순으로 평가한다.
- 경쟁이 높다는 이유만으로 큰 시장을 제외하지 않는다. 경쟁은 예상 점유율에 반영한다.
- 판매량, 수수료, 인증, 원가를 추측하지 않는다. 확인하지 못한 값은 `UNKNOWN`으로 남기고 통과를 확정하지 않는다.
- `확인된 값`, `관찰 신호`, `시나리오 가정`을 구분한다. 시나리오 주문수를 확인된 판매량처럼 표현하지 않는다.
- 가격, 리뷰, 배송조건, 구매 신호마다 조사시각과 실제 페이지 URL을 남긴다.
- 공급처마다 원문에서 단가, MOQ, 구매단위, 주문 배송비, 조사시각을 확인하고 `supplier_terms.verified=true`로 기록한다. 목록 단가를 판매 묶음 원가로 사용하지 않는다.
- `판매 묶음 원가 = 검증 단가 × 판매 묶음 수량 + 주문 배송비 ÷ 매입 수량 × 판매 묶음 수량`으로 계산한다. 판매 묶음 수량과 매입 수량이 없으면 가격 검토를 차단한다.
- 쿠팡 정상가·할인 참고가는 비교가격으로 사용하지 않는다. 검색 카드에서 의미가 확인된 할인 후 현재 실판매가만 사용하며, 가격 의미가 모호하거나 판매 묶음 수량이 다르면 제외한다. 최근 구매 1건 이상 또는 같은 판매상품 리뷰 5개 이상인 판매 근거 행만 가격 분포에 포함한다.
- 가격보다 먼저 공급처와 쿠팡 카드의 이미지·구조·규격·모델·고유 문구를 대조해 완전 동일성을 잠근다. 완전 동일 상품의 동일 묶음 가격만 직접 가격 제약으로 사용하고, 다른 상품 가격은 수요·가격 수용성의 보조 맥락으로만 기록하며 단독 탈락 근거로 쓰지 않는다.
- 시장 중앙값은 등록된 현재가 전체가 아니라 판매·수요 근거가 확인된 현재가로만 계산한다. 최근 구매 수 1건 이상을 우선 근거로 인정하고, 해당 신호가 없으면 같은 판매상품 리뷰 5개 이상을 구매 발생 대리 신호로 인정한다. 두 근거가 모두 없는 저가·고가 등록은 중앙값에서 제외하며, 수요 근거 가격이 5건 미만이면 `PRICE_REVIEW_BLOCKED`로 둔다. 리뷰는 현재 가격에서의 판매량 확정값이 아님을 보고서에 명시한다.
- 후보별 쿠팡 근거 상품 URL을 최소 5개 제공한다. 5개를 확보하지 못하면 판매상품 검색 키워드, 쿠팡 검색 URL, 관찰 상품 수 5개 이상의 검색 증거를 제공한다.
- 매 조사 라운드가 끝날 때 HTML 보고서를 갱신한다. 합격 상품뿐 아니라 `WATCH`, `REJECT`, `BLOCKED`, 접근 제한 후보를 모두 보존하고 판정 사유, 도매 URL, 쿠팡 검색어·검색 URL·상품 근거, 조사 라운드와 재개 조건을 표시한다. 사용자는 이 전체 근거를 보고 진입 상품을 선택한다.
- 로켓그로스 선소싱 마진은 원가·입출고/배송비·판매수수료·부가세를 차감해 계산한다. 정상가 40% 이상이고 10% 하락 후 30% 이상이면 `STANDARD`, 표준에 미달해도 정상가 35% 이상이고 10% 하락 후 25% 이상이면 `CONDITIONAL_TEST_PRICE_REVIEW`로 분리한다. 광고비와 실제 반품비는 판매 후 순수익 평가로 분리한다.
- 조건부 후보는 사용자 가격 승인과 실물·권리·가격 수용성 확인 전 `SHORTLIST`나 상세페이지 핸드오프로 자동 승격하지 않는다.
- 광고와 일반 결과를 구분하고 동일 상품은 판매자별 행을 보존한 뒤 상품군으로 묶어 요약한다.
- 로그인된 로컬 브라우저 세션은 사용자가 허용한 범위에서만 사용한다. 비밀번호, MFA, 동의, 계정 선택이 나오면 멈춘다.
- 유료 원격 브라우저는 사용자가 명시적으로 요청하거나 승인한 경우에만 시작한다.

## 실행 순서

1. [입력 계약](references/input-contract.md)을 읽고 실행 입력과 비용 기본값을 확정한다. 값이 없으면 기본값으로 위장하지 말고 `null`로 둔다.
2. 설치된 `browser-harness`와 `browser-use` 스킬을 읽고 그 지침을 따른다. 도매 원문은 `scripts/run_headless_browser_harness.py`가 만든 임시 프로필의 로컬 headless Chrome에서 Browser Harness로 확인한다. 쿠팡 공개 검색은 `scripts/collect_coupang_nodriver.py`의 headless 세션으로 수집한다. 둘 다 사용자의 기존 Chrome에 연결하거나 표시형 브라우저로 자동 전환하지 않는다.
3. 기본 후보 풀은 도매꾹 Best의 카테고리별 TOP 150으로 한다. 실행 단위 카테고리는 선택 입력이다. 사용자가 고르면 해당 대분류를 우선하고, 비워 두면 `전체`, `패션잡화/화장품`, `의류/언더웨어`, `출산/유아동/완구`, `가구/생활/취미`, `스포츠/건강/식품`, `가전/휴대폰/산업`을 순환한다. 기본 30개까지 수집하고 공급가, MOQ, 구매단위, 주문 배송비, 판매·매입 수량, 옵션, 공급사, 규제·인증 상태, 이미지 사용 가능성, 공급 안정성을 기록한다. 각 조건은 상품 원문 URL과 조사시각으로 검증한다. Best 순위는 수요 확정값이 아니라 도매시장 후보 발굴 신호로만 사용한다.
   목록이 크면 순위 상위만 자르지 말고 다음 명령으로 상·중·하 순위 구간과 상품명 다양성을 반영한 조사 표본을 만든다.

```powershell
python <skill-dir>/scripts/sample_top150.py `
  --input <run-dir>/domeggook-pool.json `
  --output <run-dir>/sampling-plan.json --target 30
```
   공급조건 입력은 가격 계산 전에 다음 명령으로 검증한다. 오류가 하나라도 있으면 해당 후보는 `PRICE_REVIEW_BLOCKED`다.

```powershell
python <skill-dir>/scripts/evidence_contract.py `
  --input <run-dir>/supplier-candidates.json `
  --output <run-dir>/supplier-evidence-check.json
```
4. [조사 프로토콜](references/research-protocol.md)에 따라 도매 하드 필터를 적용한다. 하드 필터 실패 후보도 삭제하지 말고 실패 사유와 URL을 남긴다.
5. 남은 후보마다 대표 카테고리, 핵심 소구, 동일상품 탐색 검색어를 만든다. 쿠팡 랭킹순 첫 화면에서 검색어별 최대 20개를 조사한다.
6. 대표 검색과 세부 검색에서 반복되는 노출, 최근 구매·리뷰 신호, 다수 판매자의 판매 흔적, 동일상품 포화도, 가격대와 브랜드 장악도를 정리한다.
7. [평가 정책](references/evaluation-policy.md)에 따라 관찰 점수와 모든 근거를 입력한다. 시장 주문수는 공개 근거로 범위를 만들 수 있을 때만 저·중·고 시나리오로 기록한다.
8. 동일·거의 동일·유사상품을 분리해 쿠팡 가격 분포를 만든다. 검색 카드의 현재 실판매가 의미가 검증되고 판매 묶음 수량이 후보와 정확히 같은 일반상품 중 최근 구매 수 1건 이상 또는 같은 판매상품 리뷰 5개 이상인 가격만 포함한다. 정상가·취소선 가격·의미가 모호한 복수 가격과 판매 근거 없는 등록가는 제외한다. 먼저 이미지·구조·규격·모델·고유 문구로 `identical`을 확정하고 동일 상품 가격은 직접 가격 제약으로 사용한다. 완전 동일품이 없으면 유사상품 가격은 별도 맥락으로만 남기고 그 가격만으로 후보를 탈락시키지 않는다. 수요 근거가 있는 독립 가격 표본 5개 미만이면 가격 검토를 차단한다. 검증된 묶음 원가와 판매 근거 가격의 중앙값을 교차해 진입가·추천가·프리미엄가를 결정하며, 추천가와 10% 하락 가격이 표준 40%·30% 또는 조건부 35%·25% 게이트를 통과해야 한다.
   가격 조사 입력이 준비되면 다음 명령으로 가격 검토 대상을 먼저 계산한다.

```powershell
python <skill-dir>/scripts/recommend_prices.py `
  --input <run-dir>/pricing-input.json `
  --output <run-dir>/price-recommendations.json
```
9. 다음 명령으로 경제성, 점수, 판정, 기대이익 범위를 계산한다.

```powershell
python <skill-dir>/scripts/evaluate_candidates.py `
  --input <run-dir>/sourcing-input.json `
  --output-dir <run-dir>/output
```

10. 다음 명령으로 결과 계약과 SHORTLIST 게이트를 검증한다.

```powershell
python <skill-dir>/scripts/validate_sourcing_output.py `
  --output-dir <run-dir>/output --strict
```

11. 검증이 실패하면 실패한 입력·근거·계산만 보완하고 다시 실행한다. 조사하지 않은 값을 채워 통과시키지 않는다.
12. 우리의 수요·마진·경쟁·운영 기준을 모두 통과한 상품이 누적 5개가 될 때까지 카테고리를 순환한다. 실행기는 `run-state.json`에 카테고리 커서, 조사 상품 ID, 회차 이력과 누적 통과 수를 저장한다. 각 수집기는 `SOURCING_CATEGORY`, `SOURCING_ROUND`, `SOURCING_ROUND_OUTPUT` 환경변수를 받아 회차별 후보 JSON을 만들고, headless Browser Harness 실행기가 `finally`에서 현재 회차의 탭·프로세스·임시 프로필을 정리한다.

누적 통과 상품이 5개보다 적으면 다음 카테고리·순위·원산지 표본을 자동 조사한다. 최대 회차, 풀 소진, CAPTCHA, 로그인 요구, 사이트 차단 또는 판매자 확인 불가에서만 안전 중단한다. 판매자로켓은 진입 가능하다. 판매량순 상위 10개에서 일반 로켓이 3개 이하면 통과 가능하고 4개 이상이면 경쟁 과다로 탈락시킨다.

표준 40%/30% 기준을 충족하지 못했더라도 정상 35% 이상, 10% 가격 하락 후 25% 이상이면 항상 조건부로 평가한다. 이 결과는 `CONDITIONAL_TEST_PRICE_REVIEW`로 분리하고 표준 통과 수와 자동 `SHORTLIST`·핸드오프에 합치지 않는다. 사용자 선택 보고서에는 포함할 수 있지만 조건부임을 명시한다. 완전 동일 가격 표본이 없으면 가격 수용성 미확인, 실물·권리·주장 근거 확인을 다음 게이트로 명시한다.

```powershell
python <skill-dir>/scripts/iterate_qualified_pool.py `
  --run-dir <run-dir> `
  --goal 5 --max-rounds 30 --collector-command "<headless Browser Harness collector command>"

python <skill-dir>/scripts/build_qualified_report.py `
  --input <run-dir>/qualified-input.json `
  --output-dir <run-dir>/output --minimum 5 --round-target 5 `
  --round <completed-round> --max-rounds 30
```

`--categories`를 생략하면 도매꾹의 `전체`와 실제 6개 대분류를 순환한다. 특정 범위만 원하면 `--categories "가구/생활/취미"`처럼 명시한다.

13. 보고서 상태를 `AWAITING_USER_SELECTION`으로 두고 도매꾹 URL, 쿠팡 URL, 가격 분포, 추천가, 정상·스트레스 마진, 수요·경쟁 근거를 제시한다. 사용자가 상품과 가격안을 선택하기 전에는 상세페이지 제작으로 넘기지 않는다.
14. 사용자가 선택한 상품만 최대 5개의 `SHORTLIST`로 승격한다. 각 상품에 샘플 확인 항목, 차별화 소구 3개, 증명 장면 3개, GIF 아이디어 1개를 완성하고 [출력 계약](references/output-contract.md)에 따라 전달한다.

## Headless Browser Harness 실행 규칙

- `scripts/run_headless_browser_harness.py`로 만든 로컬 headless Chrome만 사용한다. 실행기는 `BU_CDP_URL`을 해당 임시 세션으로 고정하고 종료 시 자신이 시작한 프로세스와 프로필만 정리한다.
- headless 실행이 차단되거나 실패해도 사용자의 기존 Chrome, 화면이 보이는 Chrome 또는 유료 원격 브라우저로 자동 전환하지 않는다. 차단 상태와 재개 지점을 남기고 중단한다.
- 현재 브라우저 화면을 스크린샷으로 확인한 뒤 좌표 클릭하고 다시 스크린샷으로 확인한다.
- 첫 탐색은 `new_tab(url)`을 사용하고 이동 뒤 `wait_for_load()`를 호출한다.
- 목록 추출은 DOM이 안정적일 때 `js(...)`를 사용하되, 화면과 상세페이지를 직접 열어 표본 값을 교차 확인한다.
- 쿠팡의 차단, CAPTCHA, 로그인 벽을 우회하지 않는다. 해당 검색어를 `blocked`로 기록하고 시장조사 완료로 표시하지 않는다.
- 검색결과 제목만으로 동일상품을 확정하지 않는다. 외형, 모델명, 고유 문구, 이미지 재사용을 함께 확인한다.
- 조회 시각은 ISO 8601과 `+09:00` 같은 오프셋을 포함해 기록한다.
- 상품별 필수 정보를 확보하거나 접근 차단 상태를 기록하면 해당 조사 탭을 즉시 닫는다.

## 판정 경계

- 하드 필터 실패 또는 명시적 마진 탈락 조건이면 점수와 관계없이 `REJECT`로 둔다.
- 총점 75점 이상이어도 핵심 비용, 시장조사, 가격 근거, 수요 시나리오, 규제 검토, 콘텐츠 핸드오프 중 하나가 불완전하면 `WATCH`로 둔다.
- 총점 60~74점은 `WATCH`, 59점 이하는 `REJECT`로 둔다.
- `SHORTLIST`는 최대 5개다. 동점이면 AI 이미지·GIF로 이점을 증명하기 쉬운 후보를 먼저 둔다.
- 규제 중요 카테고리는 자동으로 적합성을 확정하지 말고 `separate_review`로 분리한다.

## 산출물

- `qualified-candidates.json`: 보고서 원자료와 `AWAITING_USER_SELECTION` 상태
