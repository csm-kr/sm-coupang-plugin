## ADR-001. 독립 스킬을 하나의 플러그인으로 배포한다

### 배경

상품 소싱과 상세페이지 제작은 입력, 도구, 검증 방식이 다르지만 하나의 판매 준비 흐름을 이룬다. 하나의 거대한 스킬로 합치면 지침이 비대해지고 각 단계의 재사용과 테스트가 어려워진다.

### 결정

배포 단위는 `coupang-commerce-automation` 플러그인으로 통합하되, 실행 책임은 독립 스킬로 분리한다. 현재는 `coupang-product-sourcing`과 `coupang-detail-page-generator`를 포함하고, 모션·HTML·게시 QA는 후속 독립 스킬로 추가한다.

### 결과

- 장점: 각 스킬의 입력·출력·테스트 경계가 선명하고 독립 실행이 가능하다.
- 비용: 스킬 간 전달 계약과 버전 호환성을 별도로 관리해야 한다.

## ADR-002. 단계형 게이트와 명시적 사용자 승인을 사용한다

### 배경

상품 선정, 가격, 브랜드명, 광고 주장은 비용과 법적·운영 위험이 큰 결정이다. 불완전한 근거로 전체 파이프라인을 자동 실행하면 잘못된 상품을 고품질 콘텐츠로 포장할 수 있다.

### 결정

파이프라인을 소싱, 사용자 선택, 제품 검증, 시장 조사, 브랜드 승인, 기획, 제작, QA, 패키징 게이트로 나눈다. 상품 및 가격 선택과 신규 브랜드명 사용은 사용자의 명시적 승인 후에만 다음 단계로 진행한다. 게이트 실패 시 가장 가까운 원인 단계로 롤백한다.

### 결과

- 장점: 비용이 큰 제작 전에 오류를 차단하고 사용자가 핵심 사업 결정을 통제한다.
- 비용: 완전 무인 실행보다 상호작용이 늘고 완료 시간이 길어질 수 있다.

## ADR-003. 근거 우선 데이터 모델을 사용한다

### 배경

공급처 설명, 경쟁상품 관찰, 이미지 추론, 사용자 확인은 신뢰 수준이 다르다. 이를 하나의 사실처럼 취급하면 허위 광고와 제품 왜곡이 발생한다.

### 결정

모든 주요 주장과 자산을 출처, 검증 상태, 허용 용도에 연결한다. 기본 증거 상태는 다음을 사용한다.

- `CONFIRMED_USER`: 사용자가 제공하고 확인한 사실
- `CONFIRMED_SOURCE`: 공급처·제조사·공식 자료에서 확인한 사실
- `OBSERVED_IMAGE`: 실제 제품 이미지에서 직접 관찰한 사실
- `OBSERVED_MARKET`: 공개 시장·리뷰에서 관찰한 신호
- `INFERRED`: 근거로부터 추론했지만 판매 주장으로 확정할 수 없는 판단
- `CONFLICT`: 출처 간 충돌
- `UNKNOWN`: 확인 불가
- `FORBIDDEN`: 게시 또는 생성에 사용할 수 없는 정보

최종 카피와 시각적 증거는 게시 가능한 상태만 참조한다. `INFERRED`, `CONFLICT`, `UNKNOWN`, `FORBIDDEN`은 사용자에게 표시할 수 있으나 확정 주장으로 승격하지 않는다.

### 결과

- 장점: 주장-출처 추적, 감사, 재검증과 안전한 카피 생성이 가능하다.
- 비용: 산출물과 검증 로직이 늘어나고 초기 자료 정리가 필요하다.

## ADR-004. 소싱과 제작 사이에 버전된 공통 전달 계약을 둔다

### 배경

현재 두 스킬은 독립적으로 동작하며, 사람이 소싱 보고서의 내용을 상세페이지 프로젝트에 옮겨야 한다. 자유 형식 문서만 전달하면 후보 ID, 가격안, 차별화 가설과 출처가 유실될 수 있다.

### 결정

후속 구현에서 버전된 `commerce-project` JSON 계약을 도입한다. 계약에는 프로젝트·후보 ID, 워크플로 상태, 공급처 URL, 가격 시나리오, 증거, 주장, 자산, 사용자 승인, 채널 설정과 게이트 결과가 포함되어야 한다. 기존 스킬 산출물은 어댑터를 통해 이 계약으로 승격한다.

계약의 최소 원칙은 다음과 같다.

- 스키마 버전을 명시한다.
- ID 기반 참조를 사용하고 자유 문구 복사를 최소화한다.
- 소싱의 차별화 포인트와 증명 장면은 `hypothesis` 상태로 전달한다.
- 사용자의 후보·가격 승인을 기록한다.
- 공급처 정보 변경 시 경제성 게이트를 다시 실행한다.

### 결과

- 장점: 단계 간 데이터 유실을 줄이고 파이프라인 자동화와 재개가 쉬워진다.
- 비용: 스키마 마이그레이션과 하위 호환성 정책이 필요하다.

## ADR-005. 반복 계산과 검증은 결정적 스크립트로 수행한다

### 배경

마진 계산, 표본 배분, 이미지 크기, 필수 파일, OCR 기대 문구와 매니페스트 검증을 자연어 판단에만 맡기면 실행마다 결과가 달라질 수 있다.

### 결정

정량 계산과 형식 검증은 Python 스크립트로 고정하고, 에이전트는 조사·해석·기획과 예외 처리를 담당한다. 스크립트는 명시적 입력 파일을 받고 기계 판독 가능한 결과와 실패 사유를 반환해야 한다.

### 결과

- 장점: 재현성과 테스트 가능성이 높아지고 회귀 오류를 줄인다.
- 비용: 정책 변경 시 스크립트와 문서·테스트를 함께 갱신해야 한다.

## ADR-006. 실제 브라우저 조사와 출처 보존을 기본으로 한다

### 배경

검색 요약만으로는 실제 상품 동일성, 가격 옵션, 리뷰 문맥과 페이지 접근 상태를 정확히 판단하기 어렵다.

### 결정

시장·경쟁·리뷰 조사에는 실제 브라우저 기반 확인을 사용하고 방문 URL, 관찰 시점, 접근 제한과 신뢰도를 남긴다. 로그인 우회, CAPTCHA 우회 및 접근 통제 회피는 하지 않는다. 접근할 수 없는 정보는 검증 불가로 기록한다.

### 결과

- 장점: 조사 근거의 품질과 감사 가능성이 높아진다.
- 비용: 사이트 변경과 접근 제한에 민감하며 실행 시간이 늘어날 수 있다.

## ADR-007. 제품 정체성 보존을 미적 완성도보다 우선한다

### 배경

생성형 이미지가 더 세련되어 보여도 실제 SKU의 형상, 구성품, 색상 또는 사용 방식을 바꾸면 반품과 신뢰 문제를 만든다.

### 결정

제품 불변 특성과 실제 제품 자산 계보를 먼저 잠근다. 제품을 보여 주는 장면은 원본 보존 또는 합성을 우선하고, 생성은 배경·분위기·비제품 보조 요소에 제한한다. 제품 생성이 허용되는 경우에도 파일럿을 먼저 만들고 동일성 QA를 통과해야 한다. 실제 증명이 필요한 장면은 실사진 또는 실촬영으로 대체한다.

### 결과

- 장점: 제품 왜곡과 반품 위험을 줄인다.
- 비용: 사용 가능한 실물 사진이 부족하면 제작이 차단되거나 추가 촬영이 필요하다.

## ADR-008. 계획과 제작을 분리하고 10장 고정 규격을 기본값으로 한다

### 배경

이미지를 먼저 생성하면 장마다 역할이 겹치고 고객 질문과 근거의 연결이 약해진다. 출력 규격이 흔들리면 채널 업로드와 QA도 복잡해진다.

### 결정

브랜드, 구매 서사, 장별 역할, 승인 카피, UI, 자산 전략을 먼저 확정한 뒤 이미지를 제작한다. 신규 프로젝트의 기준 출력은 10장, 각 800×2400px로 고정한다. 기존 프로젝트는 매니페스트에 선언된 규격을 보존한다.

### 결과

- 장점: 구매 흐름, 시각 일관성, 자동 검사가 단순해진다.
- 비용: 제품에 따라 더 짧거나 긴 구성이 적합해도 초기 버전에서는 예외 처리가 필요하다.

## ADR-009. 자동 QA와 육안 QA를 모두 통과해야 출고한다

### 배경

파일 규격과 OCR은 자동화할 수 있지만, 제품의 미세한 변형, 비상업적 조형, 의미가 어긋난 장면과 광고 위험은 기계 검사만으로 충분하지 않다.

### 결정

최종 승격 조건을 이중화한다.

1. 자동 QA: 파일 수·크기, 매니페스트, 출처 연결, OCR, 필수 문구 및 산출물 완결성
2. 육안 QA: 제품 동일성, 한글 가독성, 정렬, 장별 효용, 주장-증거 일치, 상업성 및 광고 표현

실패한 페이지와 원인 영역만 재생성하며, 수정 이력을 남긴다. 두 검사가 모두 통과하기 전에는 최종 디렉터리로 승격하지 않는다.

### 결과

- 장점: 자동화의 속도와 사람 수준의 맥락 검사를 결합한다.
- 비용: 최종 검수 시간이 필요하고 육안 판정 기준을 지속적으로 보정해야 한다.

## ADR-010. 외부 부작용이 있는 상거래 행위는 초기 범위에서 제외한다

### 배경

발주, 결제, 상품 등록, 광고 집행은 계정 권한, 금전 지출, 법적 책임과 되돌리기 어려운 외부 변경을 수반한다.

### 결정

초기 플러그인은 조사, 분석, 콘텐츠 제작, QA와 내보내기까지 지원한다. 공급처 연락·발주·결제, 판매 채널 로그인·상품 등록, 광고 집행은 수행하지 않는다. 후속 도입 시 별도 권한 모델, 사용자 확인, 감사 로그와 롤백 전략을 갖춘 ADR을 추가한다.

### 결과

- 장점: 안전한 로컬 작업과 의사결정 지원에 집중할 수 있다.
- 비용: 사용자가 최종 게시와 상거래 실행을 직접 수행해야 한다.

## ADR-011. 공급처·마켓별 브라우저 어댑터를 분리한다

### 배경

도매꾹 Best는 Browser Use에서 카테고리 탐색과 상품 URL 수집이 안정적으로 동작했지만, 쿠팡 검색은 같은 세션에서 Akamai 접근 거절이 발생했다. 반면 실제 Chrome을 표시하는 `nodriver`에서 홈 선진입 후 검색을 직렬 처리했을 때 공개 상품 근거를 수집할 수 있었다.

### 결정

도매꾹 수집은 Browser Use, 쿠팡 공개 검색 수집은 표시형 `nodriver` 어댑터를 기본으로 한다. 쿠팡 어댑터는 새 임시 세션, 홈 선진입, 직렬 검색, 결과 0개 시 1회 재시도, 실행 후 브라우저 종료를 지킨다. 재시도 후에도 결과가 없거나 CAPTCHA·로그인이 요구되면 차단 상태와 재개 위치를 저장하고 중단한다.

이 결정은 접근 통제 우회를 허용하지 않는다. 향후 사이트 정책이나 브라우저 동작이 바뀌면 공통 평가·보고 계약은 유지하고 해당 어댑터만 교체한다.

### 결과

- 장점: 사이트별 접근 특성을 격리하면서 URL·가격·리뷰·로켓 근거의 공통 출력 계약을 유지한다.
- 비용: 브라우저 런타임이 두 종류가 되고 사이트 변경 시 각 어댑터의 회귀 검증이 필요하다.

## ADR-012. 단계형 TDD 하니스와 날짜별 보고서 보관을 사용한다

### 배경

소싱·상세페이지·모션·HTML·게시 QA는 서로 다른 검증 도구와 승인 게이트가 필요하다. 문서 지침만으로는 테스트 없는 구현, 단계 건너뛰기와 실행 보고서 경로의 혼재를 일관되게 막기 어렵다.

### 결정

루트와 하위 `AGENTS.md`를 작업 라우터로 사용하고, `harness/stages.json`에 단계 경로·테스트 경로·필수 검증 명령을 선언한다. 구현 전 같은 turn의 테스트 편집을 요구하는 `PreToolUse`, 테스트 편집을 기록하는 `PostToolUse`, 단계 검증 전 종료를 막는 `Stop` 훅을 사용한다. Git 환경에는 동일 계약을 호출하는 pre-commit 훅을 둔다.

모든 최종 실행 보고서는 `reports/YYYY/YYYY-MM-DD/<run-name>/`에 보관한다. 임시 수집물은 `tmp/`에 두고 검증 후 승격한다.

이 구조는 `csm-kr/harness-framework`의 문서 정본·얇은 라우터·실행 가능한 AC·2계층 가드 원칙을 차용하되, Codex의 공식 `.codex/hooks.json` 수명주기 훅으로 구현한다.

### 결과

- 장점: TDD 순서, 단계별 완료 조건과 보고서 이력을 자동 검증할 수 있다.
- 비용: 새 훅은 사용자가 `/hooks`에서 신뢰해야 하며, 계획 단계는 테스트·검증 도구가 생기기 전까지 완료할 수 없다.

## ADR-013. 상품기획과 콘텐츠기획을 분리하고 이미지+HTML 하이브리드로 제작한다

### 배경

상품 타깃·오퍼·주장과 상세페이지의 카피·순서·자산 명세가 하나의 기획 문서에 섞이면 어느 결정을 사용자가 승인했는지 추적하기 어렵다. 또한 한글 카피를 이미지에 직접 생성하면 사소한 문구 수정에도 이미지 전체를 다시 만들고 OCR·정렬 QA를 반복해야 한다.

### 결정

워크플로 5.3부터 `output/planning/product-plan.json`과 `content-plan.json`을 분리한다. 두 계획은 각각 현재 파일의 SHA-256에 묶인 `actor_type: user` 승인 기록이 있어야 다음 단계로 진행한다. 에이전트 추천, 일괄 제작 요청 또는 침묵은 승인으로 인정하지 않는다.

시각 증거는 `output/content-assets/`의 이미지·GIF·영상으로 만들고, 헤드라인·본문·표·FAQ·주의 문구는 `output/html/detail-page.html`의 네이티브 HTML 텍스트로 조립한다. 시각 소재마다 자동·육안 QA를 통과해야 조립할 수 있으며, 조립 후에는 흐름·브랜드·반응형·접근성·광고 표현·채널 규격을 통합 QA한다. 승인·QA는 모두 대상 파일 해시에 묶어 변경 시 자동으로 오래된 상태가 되게 한다.

5.0~5.2 프로젝트의 10장 정적 이미지 계약은 하위 호환을 위해 유지한다. 쿠팡 업로드용 정적 이미지는 후속 채널 패키징에서 5.3 HTML 정본으로부터 렌더링한다.

### 결과

- 장점: 카피·순서·스타일 수정에 이미지 재생성이 필요 없고, 상품 결정·콘텐츠 결정·소재 품질·전체 품질의 책임이 분리된다.
- 장점: 사용자 승인이 정확한 계획 버전에 묶이고 변경 후 재승인·재QA가 자동 강제된다.
- 비용: HTML을 직접 지원하지 않는 채널에는 별도 정적 렌더링과 채널 QA가 필요하다.
- 비용: 소재 QA와 통합 QA 두 층을 운영해야 한다.

## ADR-014. 판매 신호에 따라 1차 간이검증과 2차 심화검증을 분리한다

### 배경

모든 소싱 후보에 다수 샘플, 내구·세척 시험, 검사서·시험성적서 확인과 상세 실험을 선행하면 판매 가능성이 확인되지 않은 상품에도 시간과 비용이 크게 든다. 반대로 검증을 모두 생략하면 경쟁사 저별점 리뷰에서 찾은 소구 가설을 자사 제품의 확정 성능처럼 오인할 수 있다.

### 결정

제품 검증을 `phase_1` 간이검증과 `phase_2` 심화검증으로 나눈다.

- 1차는 경쟁사 별점 1~3점 반복 불만에서 소구 가설을 만들고, 동일 SKU의 최소 사실과 가능한 간이 실험만 확인한 뒤 제품기획·콘텐츠기획·테스트 출시로 진행한다.
- 1차에서 실험할 수 없는 소구는 `미검증 가설`로 보존할 수 있지만, 정량 성능·효능·안전·인증·비교우위의 확정 주장에는 사용하지 않는다.
- 상품 등록에 필요한 법률·안전·필수 인증·표시사항과 SKU 동일성은 2차로 미룰 수 없다.
- 테스트 판매 후 캠페인 시작일부터 14일 이내 `판매수량 2개 이상 AND ROAS 400% 이상`을 충족한 제품만 2차 검증으로 승격한다. 실제 성과 값은 사용자가 제공한 `CONFIRMED_USER` 근거로 기록한다.
- ROAS는 `광고 귀속 매출 ÷ 광고비 × 100`이며 광고비가 0원이거나 귀속 범위가 불명확하면 통과로 판정하지 않는다.
- 캠페인 ID별로 기간·광고비·귀속 매출·판매수량·ROAS·판정을 분리한다. SKU 승격에는 사용자가 지정한 판정 캠페인을 사용하고, 여러 캠페인의 합산은 대상과 방식을 별도 승인받는다.
- 2차는 경쟁사와 자사 후기, 다수 샘플 편차, 심화 실험, 검사서·시험성적서·인증서 적용 범위, 실제 사이즈와 관리법을 확인한다.
- 2차에서 소구·근거·사실이 달라지면 상품기획, Source of Truth, 콘텐츠기획, 이미지+HTML, 상품등록 정보와 QA를 다시 승인·실행한다.

2차 진입 기준과 캠페인 판정 방식은 정책 버전으로 관리한다. 사용자가 기준을 바꾸면 과거 기록을 덮어쓰지 않고 새 버전으로 다시 판정한다.

### 결과

- 장점: 검증 비용을 판매 가능성이 확인된 상품에 집중하고 빠르게 시장 반응을 확인할 수 있다.
- 장점: 미검증 소구를 데이터로 보존하면서도 판매 주장과 분리할 수 있다.
- 비용: 1차 콘텐츠의 주장 범위가 제한되고 2차 후 기획·콘텐츠·QA를 다시 수행해야 한다.
- 위험: 판매 신호만 우선하면 필수 안전 검증이 누락될 수 있으므로 공통 최소 안전선을 별도 게이트로 강제해야 한다.

## ADR-015. 가상 원본 품질 프리뷰를 실제 제작과 격리한다

### 배경

실제 제품 단독 원본이 없으면 제품 동일성 게이트 때문에 시각 소재 제작을 시작할 수 없다. 그러나 이미지에 카피를 굽는 기존 방식과 외부 이미지·GIF·네이티브 HTML을 조립하는 새 방식의 편집성·반응형·상업적 품질을 비교하려면, 실제 SKU와 혼동되지 않는 프리뷰 경로가 필요하다.

### 결정

사용자가 범위와 목적을 명시적으로 허용한 경우에만 `concept_only` 프로토타입을 별도 폴더에 만든다.

- 생성 마스터와 생성 장면은 `trusted_for_identity=false`이며 실제 제품 동일성·치수·사양·시험·성능 근거가 아니다.
- 프로토타입 manifest에 사용자 허용 범위, 제외된 승인, 원본 계획 해시와 `promotion_status=blocked`를 기록한다.
- 모든 화면에 `연출용 콘셉트 이미지`를 표시하고 실제 사람 픽셀, 미승인 브랜드, 이미지 안 생성 카피를 사용하지 않는다.
- 장별 외부 시각 자산, 네이티브 HTML 카피, 3~6초 GIF와 정적 대체 자산을 분리한다.
- 소재 QA에서는 콘셉트 일관성과 실제 동일성을 별도 판정하고, 조립 후 360~800px 반응형·접근성·주장 범위를 통합 QA한다.
- 실제 제품 원본으로 교체하면 소재 QA와 통합 QA를 모두 다시 실행하고, 기존 프로토타입 통과를 승계하지 않는다.

구체 체크 항목과 최소 산출물은 [콘텐츠 생성·조립 QA 체크리스트](CONTENT-PRODUCTION-CHECKLIST.md)를 정본으로 사용한다.

### 결과

- 장점: 제품 근거를 훼손하지 않고 이미지+HTML·GIF 제작 방식의 품질과 편집성을 조기에 검증할 수 있다.
- 장점: 프롬프트·자산 계보·실패 유형·QA 구조를 스킬 입력으로 축적할 수 있다.
- 비용: 실제 자산 확보 뒤 시각 소재와 QA를 다시 수행해야 한다.
- 위험: 콘셉트 결과가 완성품으로 오인될 수 있으므로 격리 폴더·지속 고지·승격 차단을 함께 강제한다.

## ADR-016. 시각 스토리보드와 수치형 AC를 플러그인 단계 계약으로 사용한다

### 배경

HTML 줄바꿈을 검증해도 이미지가 주장과 무관하거나, `object-fit: cover`로 핵심 부위가 잘리거나, 모듈 순서가 콘텐츠기획과 달라지는 문제는 남는다. 이미지 생성 프롬프트에 제품 길이·착용 범위·필수 구조와 안전영역이 없으면 생성 품질이 레이아웃의 우연에 좌우된다. 소싱부터 게시 QA까지 하나의 거대한 스킬로 유지하면 현재 단계, 승인 책임과 실패 롤백 지점도 불명확해진다.

### 결정

콘텐츠기획 승인 뒤 제작 전에 `visual-storyboard.json`을 만든다. 각 모듈은 `claim_ids`, `evidence_ids`, `asset_ids`, 장면의 상업적 역할, 필수 시각 단서, 모바일·와이드 순서, 모델 사용 정책, 주 피사체·핵심 영역의 정규화 좌표와 구조화 ImageGen 프롬프트를 가진다. 치수 장면은 값뿐 아니라 A~E 시작점·끝점·확인 상태를 분리하고, 생성 이미지에 수치를 굽지 않은 채 HTML/SVG 레이어로 관리한다.

수치형 Acceptance Criteria는 다음 내부 운영 기준을 사용한다.

- 주장-근거 연결률 100%
- 핵심 주장 직접 시각화율 100%
- 모듈 순서 일치율 100%
- 주장-이미지 연관성 육안 점수 모듈별 80/100 이상
- 주 피사체 가시율 95% 이상
- 핵심 부위 가시율 100%
- 360px·800px 한글 줄바꿈·겹침·오버플로 오류와 strict 경고 0건

이 수치는 외부 연구의 통계가 아니라 프로젝트 회귀 판정을 위한 내부 임계값이다. 실제 Chromium/Edge 좌표를 수집해 순서·연결·크롭·타이포를 자동 검증하고, 주장-이미지 의미 연결과 상업적 완성도는 모듈 캡처를 육안 평가한다. 실패하면 해당 모듈의 스토리보드·자산·HTML 중 가장 가까운 원인 단계만 롤백한다. 완성 HTML과 아직 수행하지 않은 1차·2차 실험 계획은 별도 산출물로 유지한다.

플러그인은 `coupang-commerce-orchestrator`가 현재 단계·AC·차단 사유를 보여주고 한 번에 하나의 사용자 질문만 하는 UX를 제공한다. 전문 책임은 `coupang-product-sourcing`, `coupang-product-planning`, `coupang-content-studio`, `coupang-detail-page-generator`, `coupang-publish-qa`로 나눈다. 오케스트레이터는 승인이나 전문 판정을 대신하지 않는다.

### 결과

- 장점: 이미지가 존재하는지만 보지 않고 어떤 주장과 어떤 부위를 증명하는지 수치와 좌표로 회귀 검증할 수 있다.
- 장점: 모델·치수·GIF·HTML의 역할과 수정 범위가 분리되고 실패 모듈만 재작업할 수 있다.
- 장점: 사용자는 매 단계에서 현재 위치와 다음 한 가지 결정을 명확히 확인한다.
- 비용: 스토리보드, 브라우저 좌표 수집과 육안 점수 기록을 추가로 관리해야 한다.
- 위험: 내부 임계값이 실제 전환 성과를 보장하지 않으므로 판매 데이터에 따라 버전을 올려 재조정해야 한다.

## ADR-017. 판매 근거 가격군으로 시장 중앙값을 계산한다

### 배경

등록만 되고 실제 수요 흔적이 없는 저가 상품까지 현재가 분포에 넣으면 중앙값이 내려가고, 수요가 확인된 더 높은 가격대의 판매 가능성을 과소평가한다. 반대로 가장 비싼 판매상품 한 건만 고르면 가격 수용성을 과대평가할 수 있다. 현재 공개 화면에서 확인 가능한 판매 근거는 최근 구매 수와 누적 리뷰이며, 리뷰는 현재 판매자·현재 가격에서의 판매량을 확정하지 못한다.

### 결정

마진 판정용 `market_price_distribution`은 동일상품·동일 판매 묶음·검증된 현재 실판매가 가운데 다음 판매 근거 중 하나를 가진 독립 상품군만 포함한다.

- `recent_purchase_count>=1`: 최근 구매 근거
- `review_count>=5`: 구매 발생 대리 신호인 `review_proxy`

판매 근거가 없는 저가·고가 등록은 중앙값에서 제외하되 제외 수를 보존한다. 직접 가격 승인에는 판매 근거 가격 5건 이상을 요구하며, 부족하면 전체 등록가 중앙값이나 고가 한 건으로 대체하지 않고 `PRICE_REVIEW_BLOCKED`로 둔다. 출력에는 검증 현재가 수, 판매 근거 가격 수, 제외 수, `price_basis=demand_backed_current_sale_price`와 리뷰 대리 신호 고지를 남긴다. 유사상품 가격은 가격 수용성의 보조 맥락이며 완전 동일상품 가격을 대체하는 직접 탈락 근거가 아니다.

### 결과

- 장점: 판매 흔적 없는 저가 등록이 시장 기준 가격과 마진을 왜곡하는 문제를 줄인다.
- 장점: 판매 근거·동일성·최소 표본을 함께 강제해 임의의 고가 한 건 선택을 막는다.
- 비용: 공개 화면에서 판매 근거가 부족한 후보는 더 자주 가격 검토 차단 상태가 된다.
- 위험: 누적 리뷰는 현재 가격에서 발생한 판매를 보장하지 않으므로 최근 구매 수를 우선하고 `review_proxy`를 명시해야 한다.

### 마이그레이션

기존 등록가 전체 중앙값으로 `REJECT`된 후보는 판매 근거 가격 표본으로 재검증한다. 표본이 5건 미만이면 기존 탈락 가격 판정을 철회하고 `PRICE_REVIEW_BLOCKED`로 변경한다. 기존 필드는 보존하되 새 출력 필드가 없는 보고서는 자격 보고서로 승격하지 않는다.

## ADR-018. 로컬 React 프로젝트 대시보드와 공통 프로젝트 작업공간을 사용한다

### 배경

기존 플러그인은 단계와 승인 게이트를 대화로 설명하지만 Codex의 스킬 호출 방식에 익숙하지 않은 사용자는 현재 단계, 필요한 입력과 프로젝트별 산출물 위치를 한눈에 파악하기 어렵다. 실행 보고서는 날짜별 `reports/`, 상세페이지 작업은 `detail-page/projects/`에 있어 한 상품의 전체 여정을 프로젝트 단위로 찾기도 어렵다.

### 결정

`coupang-workflow-ui` 스킬에 React·Vite 정적 대시보드와 Python 표준 라이브러리의 `127.0.0.1` 로컬 API를 포함한다. UI는 프로젝트 목록, 9개 사용자 단계, 필수 입력, 완료 기준, 승인 게이트, 차단 사유, 폴더 지도와 다음 전문 스킬 프롬프트를 제공한다.

새 프로젝트의 정본은 `commerce-project/projects/<project-id>/project.json`과 단계별 폴더다. 날짜별 실행 보고서는 `reports/`에 계속 보관하고 프로젝트에는 상대 경로만 연결한다. 기존 `detail-page/projects/`는 자동 이동·삭제하지 않고 읽기 전용 레거시 목록으로 노출한다. UI의 완료 체크는 진행 메모이며 전문 스킬의 근거·테스트·QA 통과를 대신하지 않는다.

### 결과

- 장점: Codex 문법을 모르는 사용자도 현재 단계와 필요한 데이터를 시각적으로 이해하고 프로젝트별 재개 지점을 찾을 수 있다.
- 장점: 상품별 폴더 정본과 실행 보고서 보관 정본을 분리해 중복 파일과 서로 다른 최신본을 줄인다.
- 장점: 로컬호스트와 원자적 JSON 저장만 사용해 외부 서비스·계정 없이 동작한다.
- 비용: React 빌드 산출물과 프로젝트 상태 스키마를 함께 버전 관리해야 한다.
- 비용: 기존 상세페이지 작업은 자동 마이그레이션하지 않으므로 사용자가 새 프로젝트와 연결해야 한다.

## ADR-019. 단계 제한 Codex 자동 실행과 읽기 전용 내장 콘솔을 사용한다 — ADR-020으로 대체

### 배경

실행 문장을 복사한 뒤 별도 Codex 화면에 붙여넣는 흐름은 초보 사용자에게 UI와 실행기의 경계를 다시 학습하게 하고, 장시간 작업의 진행 상태도 대시보드에서 확인할 수 없게 한다. 반대로 브라우저에 일반 셸을 노출하거나 승인·샌드박스를 우회하면 로컬 임의 명령 실행과 과도한 권한 위험이 생긴다.

### 결정

로컬 UI 서버가 현재 단계 또는 이미 완료된 단계의 전문 스킬 프롬프트만 `codex exec --json --sandbox workspace-write --ignore-user-config --ephemeral --cd <workspace> -`로 실행한다. 인증은 유지하되 자동 실행과 무관한 전역 모델·MCP 초기화와 세션 누적은 격리한다. 프롬프트는 명령줄 문자열이 아니라 표준입력으로 전달하고 `shell=False`를 사용한다. 같은 프로젝트는 한 번에 한 실행만 허용하며, JSONL 이벤트를 서버 메모리에 최대 2,000개 보관해 React의 읽기 전용 `Codex 실행 콘솔`로 표시한다. 사용자는 해당 실행을 중지할 수 있고 서버 종료 시 서버가 시작한 자식 프로세스도 정리한다.

내장 콘솔은 일반 터미널이나 임의 셸 입력을 제공하지 않는다. `danger-full-access`, 승인 우회 옵션과 자동 권한 확대를 사용하지 않는다. 비대화형 실행에서 새 승인이 필요한 작업은 실패 이벤트를 표시한다. 실행 로그는 `project.json`이나 보고서에 자동 저장하지 않는다.

### 결과

- 장점: 복사·붙여넣기 없이 현재 단계 작업을 시작하고 진행·완료·실패·중지를 같은 화면에서 확인할 수 있다.
- 장점: 단계 잠금, 프로젝트별 동시 실행 제한, `workspace-write`와 셸 미사용으로 실행 범위를 좁힌다.
- 비용: 브라우저를 새로 열어도 같은 서버 프로세스가 살아 있을 때만 메모리 로그를 다시 볼 수 있으며 서버 재시작 후 과거 콘솔 기록은 복구되지 않는다.
- 비용: 네트워크나 추가 권한 승인이 필요한 전문 작업은 비대화형 실행에서 실패할 수 있어 사용자가 별도 Codex 세션에서 승인해야 할 수 있다.

## ADR-020. 설치된 전문 도구를 로드하고 신규 산출물로 자동 실행 성공을 판정한다

### 배경

`--ignore-user-config`로 자동 실행을 격리하면 플러그인이 설치한 전문 스킬과 Browser Use 도구도 함께 사라져, 대시보드가 요청한 고배수 소싱 스킬을 찾지 못한다. 또한 비대화형 기본 승인 정책에서는 안전한 테스트·보고서 생성 명령도 승인 대기 상태가 되어 실행되지 않을 수 있다. 실제 점검에서는 Codex 프로세스가 기존 보고서를 읽고 신규 파일을 만들지 못했는데도 종료 코드 0만으로 UI가 `완료`를 표시했다.

### 결정

자동 실행은 `codex --ask-for-approval never --search exec --json --sandbox workspace-write --ephemeral --cd <workspace> -`를 사용한다. 사용자 구성을 읽어 설치된 커머스 플러그인·Browser Use 도구를 사용할 수 있게 하되 `workspace-write`와 `never`를 명령행에서 고정한다. `never`는 샌드박스나 승인을 우회하지 않고 비대화형 승인 팝업을 만들지 않으며, 허용되지 않은 작업은 즉시 실패로 돌려준다. `danger-full-access`와 `--dangerously-bypass-approvals-and-sandbox`는 계속 금지한다.

소싱 단계는 종료 코드가 0이어도 실행 전 프로젝트 링크에 없던 `reports/` 내부 HTML 상대 경로와 같은 이름의 JSON 파일이 모두 존재하고 `project.json.links.reportRuns`에 등록되어야 성공이다. 조건을 충족하지 못하면 `artifact_validation.failed`로 기록한다. 실행 종료 뒤 UI는 프로젝트를 다시 읽어 새 링크를 표시하고, 로컬 서버는 경로 이탈을 차단한 `reports/` 파일만 열어 준다.

### 결과

- 장점: 버튼 실행이 실제 설치된 소싱·브라우저 도구를 사용할 수 있고, 승인 팝업 없이 샌드박스 안의 작업을 계속한다.
- 장점: 설명만 반환하거나 기존 보고서를 재사용한 실행이 성공으로 오인되지 않는다.
- 장점: 신규 보고서가 화면에 즉시 나타나고 사용자가 직접 열 수 있다.
- 비용: 사용자 구성의 MCP·플러그인 초기화 비용과 실패 가능성이 자동 실행에 포함된다.
- 위험: 외부 사이트 접근·CAPTCHA·로그인은 여전히 차단될 수 있다. 이 경우에도 값을 만들지 않고 신규 차단 HTML·JSON 보고서를 남겨야 한다.

## ADR-021. 공개 RDS 원칙을 따르되 독립 커머스 UI 시스템을 사용한다

### 배경

기존 프로젝트 대시보드는 짙은 녹색과 형광 노랑, 세로 단계 열과 기술 정보 카드가 강해 현재 사용자가 해야 할 한 가지 행동보다 시스템 구조가 먼저 보였다. 쿠팡 엔지니어링의 [Rocket Design System 소개](https://medium.com/coupang-engineering/introducing-coupangs-design-system-baeb117949f1)는 효율성·검증된 사용성·확장성·일관성을 핵심 원칙으로 공개하지만 실제 RDS 팔레트와 컴포넌트 사양은 내부 시스템이다. 최신 [쿠팡 미디어 자산 브랜드 가이드라인](https://news.coupang.com/media-assets-brand-guidelines-kr/)은 로고 변경과 색 구성을 포함한 브랜드 자산 요소의 복사·모방을 금지한다.

### 결정

쿠팡 로고와 다색 워드마크, 비공개 RDS 토큰은 복제하지 않는다. 대신 공개된 RDS 원칙과 쇼핑 서비스의 익숙한 정보 위계를 적용한 독립 토큰을 사용한다. 주행동은 `#2563eb`, 주의·차단은 `#e5484d`, 보조 강조는 `#f97316`, 화면 바탕은 `#f5f7fa`로 정의하고 중립 회색·흰색 표면과 조합한다.

사용자 흐름은 `현재 할 일 → 필요한 입력 → Codex 실행·로그 → 결과 확인·승인 → 다음 단계` 순서로 고정한다. 9단계는 가로 진행 내비게이션으로 표시하고 선택 단계에 `aria-current="step"`을 제공한다. 모바일에서는 현재 행동 카드를 상단에 고정하고 프로젝트 열의 고정 폭과 본문 왼쪽 여백을 해제한다. 로고로 오인될 수 있는 쿠팡 워드마크 대신 독립 `Commerce Flow` 식별자를 사용한다.

### 결과

- 장점: 현재 단계와 CTA가 파일 맵·기술 정보보다 먼저 보여 초보 사용자의 다음 행동 판단 비용을 줄인다.
- 장점: 색상·상태·간격을 토큰화해 후보 카드, 승인 게이트, 콘솔, 프로젝트 탐색기에 일관되게 확장할 수 있다.
- 장점: 360px·800px에서 전체 페이지 가로 넘침 없이 단계 탐색과 현재 행동을 유지한다.
- 비용: 공개 RDS의 정확한 사내 토큰을 재현하지 않으므로 쿠팡의 공식 제품 UI와 동일하지 않다.
- 위험: 브랜드 가이드 변경 시 외부 표장·명칭 사용 여부를 다시 검토해야 한다.

## 결정 요약

```text
독립 스킬
  → 버전된 공통 계약
  → 로컬 프로젝트 대시보드·단계별 폴더
  → 단계형 게이트와 사용자 승인
  → 근거·제품 정체성 잠금
  → 상품기획 승인 → 콘텐츠기획 승인
  → 시각 스토리보드·수치형 AC
  → 시각 소재 제작·개별 QA
  → 이미지+HTML 조립·좌표 기반 통합 QA
  → 안전한 1차 테스트 패키징
  → 1차 테스트 판매 → 판매 신호 게이트
  → 통과 제품 2차 심화검증·기획 재승인·재QA
  → 확대 패키징
```

이 결정들은 사실성, 추적성, 사용자 통제와 실패 복구를 속도보다 우선한다. 변경이 필요하면 기존 항목을 덮어쓰지 않고 후속 ADR에서 대체 사유와 마이그레이션 영향을 기록한다.

## ADR 변경 규칙

- 기존 결정을 단순 삭제하거나 문맥 없이 수정하지 않는다.
- 방향이 바뀌면 새로운 ADR을 추가하고 기존 ADR 상태를 `Superseded`로 바꾼다.
- 실험 중인 제안은 `Proposed`, 승인된 결정은 `Accepted`, 폐기한 결정은 `Rejected`로 표시한다.
- 데이터 계약이나 파일 구조를 바꾸는 ADR에는 마이그레이션과 하위 호환성 영향을 기록한다.
- 구현 완료 여부는 ADR 상태와 분리해 `Implemented`, `Partial`, `Planned`로 관리한다.
