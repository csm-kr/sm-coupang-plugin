## ADR-001. 독립 스킬을 하나의 플러그인으로 배포한다

### 배경

상품 소싱과 상세페이지 제작은 입력, 도구, 검증 방식이 다르지만 하나의 판매 준비 흐름을 이룬다. 하나의 거대한 스킬로 합치면 지침이 비대해지고 각 단계의 재사용과 테스트가 어려워진다.

### 결정

배포 단위는 `coupang-commerce-automation` 플러그인으로 통합하되, 실행 책임은 독립 스킬로 분리한다. 현재는 `coupang-product-sourcing`과 `coupang-detail-page-generator`를 포함하고, 모션·HTML·게시 QA는 후속 독립 스킬로 추가한다.

### 결과

- 장점: 각 스킬의 입력·출력·테스트 경계가 선명하고 독립 실행이 가능하다.
- 비용: 스킬 간 전달 계약과 버전 호환성을 별도로 관리해야 한다.

## ADR-002. 단계형 게이트와 명시적 사용자 승인을 사용한다

### 배경

상품 선정, 가격, 브랜드명, 광고 주장은 비용과 법적·운영 위험이 큰 결정이다. 불완전한 근거로 전체 파이프라인을 자동 실행하면 잘못된 상품을 고품질 콘텐츠로 포장할 수 있다.

### 결정

파이프라인을 소싱, 사용자 선택, 제품 검증, 시장 조사, 브랜드 승인, 기획, 제작, QA, 패키징 게이트로 나눈다. 상품 및 가격 선택과 신규 브랜드명 사용은 사용자의 명시적 승인 후에만 다음 단계로 진행한다. 게이트 실패 시 가장 가까운 원인 단계로 롤백한다.

### 결과

- 장점: 비용이 큰 제작 전에 오류를 차단하고 사용자가 핵심 사업 결정을 통제한다.
- 비용: 완전 무인 실행보다 상호작용이 늘고 완료 시간이 길어질 수 있다.

## ADR-003. 근거 우선 데이터 모델을 사용한다

### 배경

공급처 설명, 경쟁상품 관찰, 이미지 추론, 사용자 확인은 신뢰 수준이 다르다. 이를 하나의 사실처럼 취급하면 허위 광고와 제품 왜곡이 발생한다.

### 결정

모든 주요 주장과 자산을 출처, 검증 상태, 허용 용도에 연결한다. 기본 증거 상태는 다음을 사용한다.

- `CONFIRMED_USER`: 사용자가 제공하고 확인한 사실
- `CONFIRMED_SOURCE`: 공급처·제조사·공식 자료에서 확인한 사실
- `OBSERVED_IMAGE`: 실제 제품 이미지에서 직접 관찰한 사실
- `OBSERVED_MARKET`: 공개 시장·리뷰에서 관찰한 신호
- `INFERRED`: 근거로부터 추론했지만 판매 주장으로 확정할 수 없는 판단
- `CONFLICT`: 출처 간 충돌
- `UNKNOWN`: 확인 불가
- `FORBIDDEN`: 게시 또는 생성에 사용할 수 없는 정보

최종 카피와 시각적 증거는 게시 가능한 상태만 참조한다. `INFERRED`, `CONFLICT`, `UNKNOWN`, `FORBIDDEN`은 사용자에게 표시할 수 있으나 확정 주장으로 승격하지 않는다.

### 결과

- 장점: 주장-출처 추적, 감사, 재검증과 안전한 카피 생성이 가능하다.
- 비용: 산출물과 검증 로직이 늘어나고 초기 자료 정리가 필요하다.

## ADR-004. 소싱과 제작 사이에 버전된 공통 전달 계약을 둔다

### 배경

현재 두 스킬은 독립적으로 동작하며, 사람이 소싱 보고서의 내용을 상세페이지 프로젝트에 옮겨야 한다. 자유 형식 문서만 전달하면 후보 ID, 가격안, 차별화 가설과 출처가 유실될 수 있다.

### 결정

후속 구현에서 버전된 `commerce-project` JSON 계약을 도입한다. 계약에는 프로젝트·후보 ID, 워크플로 상태, 공급처 URL, 가격 시나리오, 증거, 주장, 자산, 사용자 승인, 채널 설정과 게이트 결과가 포함되어야 한다. 기존 스킬 산출물은 어댑터를 통해 이 계약으로 승격한다.

계약의 최소 원칙은 다음과 같다.

- 스키마 버전을 명시한다.
- ID 기반 참조를 사용하고 자유 문구 복사를 최소화한다.
- 소싱의 차별화 포인트와 증명 장면은 `hypothesis` 상태로 전달한다.
- 사용자의 후보·가격 승인을 기록한다.
- 공급처 정보 변경 시 경제성 게이트를 다시 실행한다.

### 결과

- 장점: 단계 간 데이터 유실을 줄이고 파이프라인 자동화와 재개가 쉬워진다.
- 비용: 스키마 마이그레이션과 하위 호환성 정책이 필요하다.

## ADR-005. 반복 계산과 검증은 결정적 스크립트로 수행한다

### 배경

마진 계산, 표본 배분, 이미지 크기, 필수 파일, OCR 기대 문구와 매니페스트 검증을 자연어 판단에만 맡기면 실행마다 결과가 달라질 수 있다.

### 결정

정량 계산과 형식 검증은 Python 스크립트로 고정하고, 에이전트는 조사·해석·기획과 예외 처리를 담당한다. 스크립트는 명시적 입력 파일을 받고 기계 판독 가능한 결과와 실패 사유를 반환해야 한다.

### 결과

- 장점: 재현성과 테스트 가능성이 높아지고 회귀 오류를 줄인다.
- 비용: 정책 변경 시 스크립트와 문서·테스트를 함께 갱신해야 한다.

## ADR-006. 실제 브라우저 조사와 출처 보존을 기본으로 한다

### 배경

검색 요약만으로는 실제 상품 동일성, 가격 옵션, 리뷰 문맥과 페이지 접근 상태를 정확히 판단하기 어렵다.

### 결정

시장·경쟁·리뷰 조사에는 실제 브라우저 기반 확인을 사용하고 방문 URL, 관찰 시점, 접근 제한과 신뢰도를 남긴다. 로그인 우회, CAPTCHA 우회 및 접근 통제 회피는 하지 않는다. 접근할 수 없는 정보는 검증 불가로 기록한다.

### 결과

- 장점: 조사 근거의 품질과 감사 가능성이 높아진다.
- 비용: 사이트 변경과 접근 제한에 민감하며 실행 시간이 늘어날 수 있다.

## ADR-007. 제품 정체성 보존을 미적 완성도보다 우선한다

### 배경

생성형 이미지가 더 세련되어 보여도 실제 SKU의 형상, 구성품, 색상 또는 사용 방식을 바꾸면 반품과 신뢰 문제를 만든다.

### 결정

제품 불변 특성과 실제 제품 자산 계보를 먼저 잠근다. 제품을 보여 주는 장면은 원본 보존 또는 합성을 우선하고, 생성은 배경·분위기·비제품 보조 요소에 제한한다. 제품 생성이 허용되는 경우에도 파일럿을 먼저 만들고 동일성 QA를 통과해야 한다. 실제 증명이 필요한 장면은 실사진 또는 실촬영으로 대체한다.

### 결과

- 장점: 제품 왜곡과 반품 위험을 줄인다.
- 비용: 사용 가능한 실물 사진이 부족하면 제작이 차단되거나 추가 촬영이 필요하다.

## ADR-008. 계획과 제작을 분리하고 10장 고정 규격을 기본값으로 한다

### 배경

이미지를 먼저 생성하면 장마다 역할이 겹치고 고객 질문과 근거의 연결이 약해진다. 출력 규격이 흔들리면 채널 업로드와 QA도 복잡해진다.

### 결정

브랜드, 구매 서사, 장별 역할, 승인 카피, UI, 자산 전략을 먼저 확정한 뒤 이미지를 제작한다. 신규 프로젝트의 기준 출력은 10장, 각 800×2400px로 고정한다. 기존 프로젝트는 매니페스트에 선언된 규격을 보존한다.

### 결과

- 장점: 구매 흐름, 시각 일관성, 자동 검사가 단순해진다.
- 비용: 제품에 따라 더 짧거나 긴 구성이 적합해도 초기 버전에서는 예외 처리가 필요하다.

## ADR-009. 자동 QA와 육안 QA를 모두 통과해야 출고한다

### 배경

파일 규격과 OCR은 자동화할 수 있지만, 제품의 미세한 변형, 비상업적 조형, 의미가 어긋난 장면과 광고 위험은 기계 검사만으로 충분하지 않다.

### 결정

최종 승격 조건을 이중화한다.

1. 자동 QA: 파일 수·크기, 매니페스트, 출처 연결, OCR, 필수 문구 및 산출물 완결성
2. 육안 QA: 제품 동일성, 한글 가독성, 정렬, 장별 효용, 주장-증거 일치, 상업성 및 광고 표현

실패한 페이지와 원인 영역만 재생성하며, 수정 이력을 남긴다. 두 검사가 모두 통과하기 전에는 최종 디렉터리로 승격하지 않는다.

### 결과

- 장점: 자동화의 속도와 사람 수준의 맥락 검사를 결합한다.
- 비용: 최종 검수 시간이 필요하고 육안 판정 기준을 지속적으로 보정해야 한다.

## ADR-010. 외부 부작용이 있는 상거래 행위는 초기 범위에서 제외한다

### 배경

발주, 결제, 상품 등록, 광고 집행은 계정 권한, 금전 지출, 법적 책임과 되돌리기 어려운 외부 변경을 수반한다.

### 결정

초기 플러그인은 조사, 분석, 콘텐츠 제작, QA와 내보내기까지 지원한다. 공급처 연락·발주·결제, 판매 채널 로그인·상품 등록, 광고 집행은 수행하지 않는다. 후속 도입 시 별도 권한 모델, 사용자 확인, 감사 로그와 롤백 전략을 갖춘 ADR을 추가한다.

### 결과

- 장점: 안전한 로컬 작업과 의사결정 지원에 집중할 수 있다.
- 비용: 사용자가 최종 게시와 상거래 실행을 직접 수행해야 한다.

## ADR-011. 공급처·마켓별 브라우저 어댑터를 분리한다

### 배경

도매꾹 Best는 Browser Use에서 카테고리 탐색과 상품 URL 수집이 안정적으로 동작했지만, 쿠팡 검색은 같은 세션에서 Akamai 접근 거절이 발생했다. 반면 실제 Chrome을 표시하는 `nodriver`에서 홈 선진입 후 검색을 직렬 처리했을 때 공개 상품 근거를 수집할 수 있었다.

### 결정

도매꾹 수집은 Browser Use, 쿠팡 공개 검색 수집은 표시형 `nodriver` 어댑터를 기본으로 한다. 쿠팡 어댑터는 새 임시 세션, 홈 선진입, 직렬 검색, 결과 0개 시 1회 재시도, 실행 후 브라우저 종료를 지킨다. 재시도 후에도 결과가 없거나 CAPTCHA·로그인이 요구되면 차단 상태와 재개 위치를 저장하고 중단한다.

이 결정은 접근 통제 우회를 허용하지 않는다. 향후 사이트 정책이나 브라우저 동작이 바뀌면 공통 평가·보고 계약은 유지하고 해당 어댑터만 교체한다.

### 결과

- 장점: 사이트별 접근 특성을 격리하면서 URL·가격·리뷰·로켓 근거의 공통 출력 계약을 유지한다.
- 비용: 브라우저 런타임이 두 종류가 되고 사이트 변경 시 각 어댑터의 회귀 검증이 필요하다.

## ADR-012. 단계형 TDD 하니스와 날짜별 보고서 보관을 사용한다

### 배경

소싱·상세페이지·모션·HTML·게시 QA는 서로 다른 검증 도구와 승인 게이트가 필요하다. 문서 지침만으로는 테스트 없는 구현, 단계 건너뛰기와 실행 보고서 경로의 혼재를 일관되게 막기 어렵다.

### 결정

루트와 하위 `AGENTS.md`를 작업 라우터로 사용하고, `harness/stages.json`에 단계 경로·테스트 경로·필수 검증 명령을 선언한다. 구현 전 같은 turn의 테스트 편집을 요구하는 `PreToolUse`, 테스트 편집을 기록하는 `PostToolUse`, 단계 검증 전 종료를 막는 `Stop` 훅을 사용한다. Git 환경에는 동일 계약을 호출하는 pre-commit 훅을 둔다.

모든 최종 실행 보고서는 `reports/YYYY/YYYY-MM-DD/<run-name>/`에 보관한다. 임시 수집물은 `tmp/`에 두고 검증 후 승격한다.

이 구조는 `csm-kr/harness-framework`의 문서 정본·얇은 라우터·실행 가능한 AC·2계층 가드 원칙을 차용하되, Codex의 공식 `.codex/hooks.json` 수명주기 훅으로 구현한다.

### 결과

- 장점: TDD 순서, 단계별 완료 조건과 보고서 이력을 자동 검증할 수 있다.
- 비용: 새 훅은 사용자가 `/hooks`에서 신뢰해야 하며, 계획 단계는 테스트·검증 도구가 생기기 전까지 완료할 수 없다.

## ADR-013. 상품기획과 콘텐츠기획을 분리하고 이미지+HTML 하이브리드로 제작한다

### 배경

상품 타깃·오퍼·주장과 상세페이지의 카피·순서·자산 명세가 하나의 기획 문서에 섞이면 어느 결정을 사용자가 승인했는지 추적하기 어렵다. 또한 한글 카피를 이미지에 직접 생성하면 사소한 문구 수정에도 이미지 전체를 다시 만들고 OCR·정렬 QA를 반복해야 한다.

### 결정

워크플로 5.3부터 `output/planning/product-plan.json`과 `content-plan.json`을 분리한다. 두 계획은 각각 현재 파일의 SHA-256에 묶인 `actor_type: user` 승인 기록이 있어야 다음 단계로 진행한다. 에이전트 추천, 일괄 제작 요청 또는 침묵은 승인으로 인정하지 않는다.

시각 증거는 `output/content-assets/`의 이미지·GIF·영상으로 만들고, 헤드라인·본문·표·FAQ·주의 문구는 `output/html/detail-page.html`의 네이티브 HTML 텍스트로 조립한다. 시각 소재마다 자동·육안 QA를 통과해야 조립할 수 있으며, 조립 후에는 흐름·브랜드·반응형·접근성·광고 표현·채널 규격을 통합 QA한다. 승인·QA는 모두 대상 파일 해시에 묶어 변경 시 자동으로 오래된 상태가 되게 한다.

5.0~5.2 프로젝트의 10장 정적 이미지 계약은 하위 호환을 위해 유지한다. 쿠팡 업로드용 정적 이미지는 후속 채널 패키징에서 5.3 HTML 정본으로부터 렌더링한다.

### 결과

- 장점: 카피·순서·스타일 수정에 이미지 재생성이 필요 없고, 상품 결정·콘텐츠 결정·소재 품질·전체 품질의 책임이 분리된다.
- 장점: 사용자 승인이 정확한 계획 버전에 묶이고 변경 후 재승인·재QA가 자동 강제된다.
- 비용: HTML을 직접 지원하지 않는 채널에는 별도 정적 렌더링과 채널 QA가 필요하다.
- 비용: 소재 QA와 통합 QA 두 층을 운영해야 한다.

## ADR-014. 판매 신호에 따라 1차 간이검증과 2차 심화검증을 분리한다

### 배경

모든 소싱 후보에 다수 샘플, 내구·세척 시험, 검사서·시험성적서 확인과 상세 실험을 선행하면 판매 가능성이 확인되지 않은 상품에도 시간과 비용이 크게 든다. 반대로 검증을 모두 생략하면 경쟁사 저별점 리뷰에서 찾은 소구 가설을 자사 제품의 확정 성능처럼 오인할 수 있다.

### 결정

제품 검증을 `phase_1` 간이검증과 `phase_2` 심화검증으로 나눈다.

- 1차는 경쟁사 별점 1~3점 반복 불만에서 소구 가설을 만들고, 동일 SKU의 최소 사실과 가능한 간이 실험만 확인한 뒤 제품기획·콘텐츠기획·테스트 출시로 진행한다.
- 1차에서 실험할 수 없는 소구는 `미검증 가설`로 보존할 수 있지만, 정량 성능·효능·안전·인증·비교우위의 확정 주장에는 사용하지 않는다.
- 상품 등록에 필요한 법률·안전·필수 인증·표시사항과 SKU 동일성은 2차로 미룰 수 없다.
- 테스트 판매 후 캠페인 시작일부터 14일 이내 `판매수량 2개 이상 AND ROAS 400% 이상`을 충족한 제품만 2차 검증으로 승격한다. 실제 성과 값은 사용자가 제공한 `CONFIRMED_USER` 근거로 기록한다.
- ROAS는 `광고 귀속 매출 ÷ 광고비 × 100`이며 광고비가 0원이거나 귀속 범위가 불명확하면 통과로 판정하지 않는다.
- 캠페인 ID별로 기간·광고비·귀속 매출·판매수량·ROAS·판정을 분리한다. SKU 승격에는 사용자가 지정한 판정 캠페인을 사용하고, 여러 캠페인의 합산은 대상과 방식을 별도 승인받는다.
- 2차는 경쟁사와 자사 후기, 다수 샘플 편차, 심화 실험, 검사서·시험성적서·인증서 적용 범위, 실제 사이즈와 관리법을 확인한다.
- 2차에서 소구·근거·사실이 달라지면 상품기획, Source of Truth, 콘텐츠기획, 이미지+HTML, 상품등록 정보와 QA를 다시 승인·실행한다.

2차 진입 기준과 캠페인 판정 방식은 정책 버전으로 관리한다. 사용자가 기준을 바꾸면 과거 기록을 덮어쓰지 않고 새 버전으로 다시 판정한다.

### 결과

- 장점: 검증 비용을 판매 가능성이 확인된 상품에 집중하고 빠르게 시장 반응을 확인할 수 있다.
- 장점: 미검증 소구를 데이터로 보존하면서도 판매 주장과 분리할 수 있다.
- 비용: 1차 콘텐츠의 주장 범위가 제한되고 2차 후 기획·콘텐츠·QA를 다시 수행해야 한다.
- 위험: 판매 신호만 우선하면 필수 안전 검증이 누락될 수 있으므로 공통 최소 안전선을 별도 게이트로 강제해야 한다.

## 결정 요약

```text
독립 스킬
  → 버전된 공통 계약
  → 단계형 게이트와 사용자 승인
  → 근거·제품 정체성 잠금
  → 상품기획 승인 → 콘텐츠기획 승인
  → 시각 소재 제작·개별 QA
  → 이미지+HTML 조립·통합 QA
  → 안전한 1차 테스트 패키징
  → 1차 테스트 판매 → 판매 신호 게이트
  → 통과 제품 2차 심화검증·기획 재승인·재QA
  → 확대 패키징
```

이 결정들은 사실성, 추적성, 사용자 통제와 실패 복구를 속도보다 우선한다. 변경이 필요하면 기존 항목을 덮어쓰지 않고 후속 ADR에서 대체 사유와 마이그레이션 영향을 기록한다.

## ADR 변경 규칙

- 기존 결정을 단순 삭제하거나 문맥 없이 수정하지 않는다.
- 방향이 바뀌면 새로운 ADR을 추가하고 기존 ADR 상태를 `Superseded`로 바꾼다.
- 실험 중인 제안은 `Proposed`, 승인된 결정은 `Accepted`, 폐기한 결정은 `Rejected`로 표시한다.
- 데이터 계약이나 파일 구조를 바꾸는 ADR에는 마이그레이션과 하위 호환성 영향을 기록한다.
- 구현 완료 여부는 ADR 상태와 분리해 `Implemented`, `Partial`, `Planned`로 관리한다.
