## 1. 제품 개요

`coupang-commerce-automation`은 상품 후보 발굴부터 시장성·수익성 검증, 사용자 상품 선택, 제품 사실 검증, 상품기획·콘텐츠기획 승인, 시각 소재 제작, 편집 가능한 HTML 조립 및 QA까지 연결하는 Codex 플러그인이다.

현재 배포 단위에는 다음 여덟 스킬이 포함된다.

- `coupang-product-sourcing`: 도매 공급처 후보 조사, 쿠팡 시장 검증, 가격·마진 평가, 후보 선별 및 HTML 보고서 생성
- `coupang-best-high-markup-sourcing`: 도매꾹 Best 층화 표본에서 사용자가 지정한 개당 공급가 상한·최소 가격 배수와 판매 근거를 만족하는 쿠팡 동일 1개 상품 pair를 찾고 일반 소싱 게이트로 전달
- `coupang-commerce-orchestrator`: 현재 단계·AC·차단 사유와 다음 사용자 질문을 관리
- `coupang-workflow-ui`: 프로젝트 생성·목록·폴더 지도·현재 단계·필수 입력·승인 게이트를 제공하고 버튼 한 번으로 현재 Codex 작업을 실행해 브라우저 콘솔에 표시
- `coupang-product-planning`: 저평점 리뷰, SKU 해결 가능성, 근거와 1차·2차 실험을 상품기획으로 변환
- `coupang-content-studio`: 주장-근거-장면 스토리보드, ImageGen 프롬프트와 소재 QA를 관리
- `coupang-detail-page-generator`: 외부 시각 자산과 네이티브 HTML을 조립하고 브라우저 QA를 수행
- `coupang-publish-qa`: 모듈 순서·크롭·주장 연결·타이포·접근성·광고 표현을 게시 전에 판정

제품의 핵심은 단순 콘텐츠 생성이 아니라, 모든 추천과 표현을 출처 및 검증 상태에 연결하는 근거 기반 커머스 제작 파이프라인이다.

## 2. 문제 정의

온라인 판매 준비 과정은 소싱, 시장 조사, 마진 계산, 상품 선정, 상세페이지 제작이 서로 분리되어 있다. 이로 인해 다음 문제가 반복된다.

- 상위 노출 또는 낮은 공급가만 보고 상품을 선택해 실제 수요와 경쟁 강도를 놓친다.
- 공급가, 수수료, 물류비, 부가세를 일관되게 반영하지 않아 기대 마진과 실제 마진이 달라진다.
- 공급처 정보, 실물 사진, 경쟁상품 정보가 섞이며 제품의 형태·기능·구성 또는 성능 표현이 왜곡된다.
- 경쟁 리뷰에서 발견한 고객 불안을 상세페이지의 주장과 증거로 연결하지 못한다.
- 생성형 이미지의 한글 오류, 제품 변형, 비상업적 레이아웃, 과장 광고를 출고 전에 체계적으로 잡지 못한다.
- 소싱 결과가 상세페이지 단계로 구조화되어 전달되지 않아 조사와 판단을 반복한다.
- Codex 스킬 이름과 프롬프트 문법을 모르는 사용자는 현재 단계·필수 입력·승인 책임을 파악하기 어렵다.
- 날짜별 보고서와 상세페이지 작업 폴더가 분리되어 한 상품의 전체 진행 상태와 최신 산출물을 찾기 어렵다.

## 3. 제품 목표

### 3.1 핵심 목표

1. 공개된 공급처 및 시장 근거를 바탕으로 판매 가능성이 높은 후보를 최소 5개 제안한다.
2. 정상 판매가 40% 이상·10% 하락 시 30% 이상을 표준 통과선으로, 정상 판매가 35% 이상·10% 하락 시 25% 이상을 사용자 검토가 필요한 조건부 통과선으로 검증한다.
3. 사용자가 상품과 가격안을 명시적으로 선택한 이후에만 상세페이지 제작 단계로 진행한다.
4. 제품 사실, 주장, 이미지, 출처의 계보를 유지해 허위·추정 표현과 제품 왜곡을 차단한다.
5. 경쟁상품 및 리뷰의 고객 불안을 구매 질문, 증거 장면, 카피와 연결한다.
6. 상품기획과 콘텐츠기획을 분리하고 각각 현재 버전을 사용자가 승인하게 한다.
7. 시각 자산과 HTML 텍스트를 분리해 카피·순서·스타일 수정 시 이미지 재생성을 최소화한다.
8. 소재별 QA와 조립 후 통합 QA를 모두 사용해 출고 가능한 결과물만 승격한다.
9. 1차 간이검증으로 시장 반응을 빠르게 확인하고, 판매 신호를 통과한 제품에만 2차 심화검증 비용을 투입한다.
10. 프로젝트별 현재 단계·필수 입력·승인·차단·폴더·보고서 링크를 브라우저 UI에서 한눈에 확인한다.
11. 실행 문장을 복사·붙여넣지 않고 현재 단계의 Codex 작업을 버튼으로 시작하고 같은 화면에서 진행 로그와 종료 상태를 확인한다.

## 4. 사용자와 핵심 사용 시나리오

### 4.1 주요 사용자

- 신규 판매자: 무엇을 판매할지와 적정 판매가를 결정하려는 사용자
- 운영 판매자: 신규 SKU를 빠르게 검증하고 상세페이지 제작 시간을 줄이려는 사용자
- 상품기획자: 공급처·경쟁·리뷰 근거를 한 흐름으로 관리하려는 사용자
- 콘텐츠 제작자: 제품 사실과 승인 카피가 잠긴 제작 명세가 필요한 사용자

### 4.2 핵심 흐름

1. 사용자가 브라우저 대시보드에서 프로젝트를 만들면 카테고리 `전체`, 도매꾹 개당 공급가 상한 `5,000원`, 최소 가격 배수 `3배`가 기본 입력되고, 필요하면 수정한 뒤 `Codex 작업 시작`을 누른다. 시작 방식은 `도매꾹 Best 고배수 pair 탐색`으로 고정한다.
2. 시스템이 도매 후보를 층화 표본으로 조사하고 비용·수요·경쟁·운영 위험을 평가한다. 카테고리를 비우면 `전체`와 실제 6개 대분류를 순환한다.
3. 실제 조사 제품 또는 검증 차단 사유를 도매꾹↔쿠팡 pair URL, 공급가, 현재 실판매가, 배수와 리뷰·만족 라벨 근거가 담긴 HTML·JSON 보고서로 먼저 제시하고, 전체 검증 결과의 가격안과 정상·하락 마진은 별도로 제시한다.
4. 시스템은 `AWAITING_USER_SELECTION` 상태에서 멈춘다.
5. 사용자가 상품과 가격안을 승인한다.
6. 시스템이 동일 SKU의 최소 사실과 필수 법률·안전 정보를 확인하고, 가능한 간이 실험으로 1차 소구 가설을 분류한다.
7. 경쟁상품 5개 이상과 접근 가능한 저별점 리뷰를 조사해 고객 불안과 증거 요구를 구조화한다.
8. 미검증 가설을 확정 주장과 분리한 1차 상품기획을 만들고 사용자의 승인을 받는다.
9. 장별 카피·근거·시각 자산·QA 기준을 콘텐츠기획으로 만들고 사용자의 승인을 받는다.
10. 이미지·GIF·영상 소재를 생성·촬영하고 소재별 자동·육안 QA를 수행한다.
11. 승인 소재와 네이티브 HTML 카피를 조립하고 반응형·접근성·광고 표현·채널 규격 통합 QA를 수행한다.
12. 소량 테스트 판매에서 판매·전환·이익·반품·문의·별점·후기를 확인한다.
13. 사용자 기준을 통과한 상품만 다수 샘플·검사서·심화 실험의 2차 검증으로 승격하고, 변경된 상품기획·콘텐츠기획을 재승인한다.

## 5. 기능 요구사항

### FR-01. 입력 계약

- 고배수 소싱 필수 입력은 목표 판매 채널, 도매꾹 개당 공급가 상한과 최소 가격 배수다. 새 프로젝트 기본값은 카테고리 `전체`, 공급가 상한 `5,000원`, 최소 배수 `3배`다. 상품 카테고리는 선택 입력이며 `전체`, `패션잡화/화장품`, `의류/언더웨어`, `출산/유아동/완구`, `가구/생활/취미`, `스포츠/건강/식품`, `가전/휴대폰/산업` 중 선택할 수 있다.
- 카테고리를 비우면 도매꾹 Best의 `전체`와 6개 대분류를 순환한다. 사용자에게 일반 소싱 모드를 별도 선택지로 노출하지 않는다.
- 상세페이지 단계에는 공급처 URL과 실제 제품을 식별할 수 있는 이미지 또는 사용자 확인이 필요하다.
- 경쟁상품, 리뷰, 참고 상세페이지, 기존 브랜드 자료는 선택 입력으로 받는다.
- 확인할 수 없는 필수 제품 정보는 추정하지 않고 `UNKNOWN` 또는 `BLOCKED`로 기록한다.

### FR-02. 상품 후보 수집과 표본화

- 도매꾹·도매매 등 허용된 공개 공급처에서 후보를 수집한다.
- 기본 탐색 모집단은 전체 또는 카테고리별 TOP 150이다.
- 기본 30개 표본을 상위 1~30위 40%, 중위 31~75위 35%, 하위 76~150위 25%로 층화한다.
- 동일 상품, 옵션 차이, 위험 카테고리 및 명백한 부적합 후보를 중복·제외 처리한다.
- 고배수 탐색 프로필은 상세 원문의 개당 단가가 사용자 상한 이하이고 판매 묶음이 1개이며, 쿠팡의 완전 동일 1개 상품 현재 실판매가가 사용자 지정 최소 가격 배수 이상인 pair를 찾는다.
- 같은 쿠팡 판매상품에 리뷰 1개 이상 또는 만족 인원 100명 이상 라벨이 있을 때 판매 근거로 인정한다. 리뷰·만족 라벨은 구매 발생 대리 신호로만 사용하고, 정상가·취소선 가격·유사상품·다른 묶음·미검증 동일성은 고배수 조건에 사용하지 않는다.
- 기준을 충족한 pair가 하나라도 있으면 `HIGH_MARKUP_DISCOVERY`이며 더 낮은 가격의 다른 쿠팡 등록은 탐색 탈락 근거로 사용하지 않는다.
- `HIGH_MARKUP_DISCOVERY`는 전체 비용·마진·수요·경쟁·운영 검증과 사용자 승인 전 `SHORTLIST`로 승격하지 않는다.
- 고배수 탐색 보고서는 `도매꾹 상품·개당 원가 ↔ 쿠팡 판매상품·현재가` pair, 가격 배수, 판매 근거와 두 URL을 기본 비교 단위로 사용한다. 전체 가격 분포와 수익률은 후속 가격 승인 검증으로 분리한다. 실패·차단 보고서에도 사람이 읽을 수 있는 실패 이유와 이번 실행의 도매꾹 표본 상품명·URL·카테고리·순위·확인 단가를 보존한다.

### FR-03. 시장·수익성 검증

- 공급가뿐 아니라 수수료, 물류비, 부가세 및 확인 가능한 비용을 반영한다.
- 쿠팡의 동일·유사 상품 가격 분포와 경쟁 밀도를 확인한다.
- 마진 판정용 시장 중앙값은 동일상품·동일 묶음·검증 현재가 중 최근 구매 수 1건 이상 또는 리뷰 5개 이상인 판매 근거 가격만으로 계산한다. 판매 근거 없는 등록가는 제외하고, 독립 표본이 5개 미만이면 가격 승인을 차단한다.
- 리뷰 수는 구매 발생 대리 신호로 표시하며 현재 판매자·현재 가격에서의 판매량 확정값으로 표현하지 않는다.
- 정상가와 10% 가격 하락 시나리오의 기여이익 및 기여이익률을 계산한다.
- 표준 통과선은 정상가 40% 이상·10% 하락 시 30% 이상이고, 조건부 통과선은 표준 미달이면서 정상가 35% 이상·10% 하락 시 25% 이상이다.
- 조건부 후보는 보고서에 명확히 구분해 포함할 수 있지만 사용자 가격 승인과 실물·권리·가격 수용성 확인 전 `SHORTLIST`나 상세페이지로 자동 승격하지 않는다.
- 가격안은 `entry`, `recommended`, `premium`, `reject`로 분류하며 근거 없는 상시 할인 표현을 만들지 않는다.

### FR-04. 후보 평가와 보고

- 수요, 경쟁, 수익성, 차별성, 운영 위험을 구조화해 평가한다.
- 후보를 `SHORTLIST`, `CONDITIONAL`, `WATCH`, `REJECT`로 분류한다.
- 기준 통과 후보가 5개 미만이면 탐색 범위를 확장하고 완료 처리하지 않는다.
- 보고서에는 후보별 도매꾹↔쿠팡 pair URL, 공급가, 현재 실판매가, 배수, 리뷰·만족 라벨, 판정 이유가 포함되어야 한다. 전체 검증 보고서에는 별도로 가격안과 정상·하락 마진을 포함한다.
- `Codex 작업 시작`은 과정 설명만 반환하지 않고 실제 제품 후보 또는 접근·근거 차단 결과가 담긴 HTML·JSON 보고서를 생성하고 프로젝트 `links.reportRuns`에 연결해야 한다.
- 소싱 종료 상태는 `AWAITING_USER_SELECTION`이어야 한다.

### FR-05. 사용자 승인과 단계 전환

- 상품 및 가격안 선택 전에는 상세페이지 제작을 시작하지 않는다.
- 승인된 후보만 다음 프로젝트로 승격한다.
- 사용자는 소싱 JSON 후보를 UI 카드에서 직접 비교·클릭하며 후보 ID·상품명·공급처 URL을 다시 입력하지 않는다.
- 전체 소싱 검증의 `SHORTLIST` 한 건과 0보다 큰 판매가를 확인한 경우에만 한 번의 확정 동작으로 승인 기록을 저장하고 상품기획 단계로 이동한다.
- `PRICE_REVIEW_BLOCKED`, `HIGH_MARKUP_DISCOVERY`, 조건부·탈락 후보는 보고서 근거로 보존하지만 소싱 주 결과 화면에는 내부 차단 코드 대신 실제 pair 또는 `조건에 맞는 pair를 찾지 못했습니다.`를 표시하며 다음 단계로 넘기지 않는다.
- 소싱의 차별화 가설과 증명 장면은 확정 사실이 아닌 가설로 전달한다.
- 공급처 정보 변경으로 수익성 기준이 깨지면 소싱 단계로 되돌린다.

### FR-06. 제품 사실과 정체성 잠금

- 입력 자산을 제품 원본, 공급처 참고, 경쟁 참고, 스타일 참고, 생성 초안, 측정 증거로 분류한다.
- 제품 외형의 비율, 전후·좌우·상하, 개구부, 손잡이, 부속품, 색상, 투명도, 라벨·각인 등 불변 특성을 기록한다.
- 제품 주장마다 출처, 검증 상태, 허용 카피 및 추가 증거 요구를 연결한다.
- 실제 제품 계보가 없는 생성 자산은 최종 제품 증거로 사용하지 않는다.
- 사람 포함 이미지 또는 제품 식별이 불가능한 자산은 안전 상태에 따라 차단한다.
- 검증 단계는 `phase_1` 또는 `phase_2`로 기록한다.
- 1차는 동일 SKU, 필수 표시·안전 정보, 확인된 사실과 가능한 간이 실험을 최소 계약으로 사용한다.
- 1차에서 실험하지 못한 소구는 `미검증 가설`로 보존하되 판매 가능한 확정 주장과 분리한다.
- 판매 가능성이 확인된 상품은 2차에서 다수 샘플 편차, 심화 실험, 검사서·시험성적서 적용 범위, 실제 사이즈·관리법을 검증한다.

### FR-07. 경쟁상품 및 리뷰 조사

- 실제 브라우저로 경쟁상품 5개 이상과 접근 가능한 리뷰를 조사한다.
- 반복 칭찬, 불만, 오해, 반품 위험, 구매 전 질문과 상세페이지 증거 방식을 기록한다.
- 관찰한 시장 정보와 자사 제품 주장을 분리한다.
- 접근 제한이나 근거 부족을 우회하거나 임의 보완하지 않고 명시한다.
- 1차에서는 경쟁사 별점 1~3점의 반복 불만을 소구 가설과 간이 실험 항목으로 변환한다.
- 경쟁사 별점 1~3점 리뷰의 URL·조사시각·표본 범위·반복 불만은 `coupang-product-planning`이 Browser Harness로 직접 조사하며 사용자에게 리뷰 URL이나 조사 파일 경로를 입력시키지 않는다.
- 2차에서는 경쟁사 최신 리뷰와 자사 초기 별점·후기·문의·반품 사유를 통합해 소구와 위험을 재판정한다.

### FR-08. 상품기획·콘텐츠기획과 사용자 승인

- 기존 브랜드가 있으면 브랜드 잠금 규칙을 보존한다.
- 기존 브랜드가 없으면 제품 사실과 고객 문제에서 제품별 브랜드 후보를 만들고 사용자의 승인을 받는다.
- 상품기획은 타깃·문제·포지셔닝·오퍼·게시 가능 주장·금지사항을 포함한다.
- 상품기획 UI는 사이즈·실측, 구성, 소재, 관리법을 각각 필수 입력으로 받고 값마다 실측·라벨·공급처 원문 중 하나 이상의 근거를 연결한다.
- 상품기획은 공급처 상세 URL 하나만으로 상세 본문·옵션·상품정보·공개 후기·문의를 먼저 조사할 수 있다. 조사 결과와 소구점 후보를 사용자에게 보고한 뒤 사용자가 판매가·묶음·판매 옵션과 소구점을 선택하는 오퍼 결정 게이트를 통과해야 최종 기획을 확정한다.
- 콘텐츠기획은 모듈별 카피·근거·시각 자산 ID·수정 가능 필드·QA 기준을 포함한다.
- 콘텐츠기획 승인 뒤 각 모듈의 주장·근거·자산·필수 시각 단서·크롭 안전영역·레이아웃 순서를 `visual-storyboard.json`으로 구체화한다.
- 두 계획은 별도 산출물이며 각각 현재 파일 해시에 묶인 명시적 사용자 승인이 필요하다.
- 장별로 구매 이유, 구매 질문, 근거 ID, 증거 유형, 제품 시점, 금지 요소와 QA 기준을 정의한다.
- 10장은 문제 인식, 핵심 효용, 기능 증거, 사용 장면, 비교, 사용·관리, FAQ, 신뢰 및 구매 유도의 흐름을 갖는다.
- 확정되지 않은 수치·성능·인증·후기·평점·판매량은 표현하지 않는다.
- 1차 상품기획은 `검증 사실`, `간이 실험 지지`, `미검증 가설`, `사용 금지`를 분리하며, 미검증 가설이 남아 있어도 확정 주장만으로 콘텐츠기획을 진행할 수 있다.
- 2차 근거로 소구·사이즈·관리법·주의사항이 바뀌면 두 계획의 기존 승인을 만료시키고 재승인을 요구한다.

### FR-09. 시각 소재와 하이브리드 HTML 제작

- 워크플로 5.3의 카피·표·FAQ·주의 문구는 네이티브 HTML로 두고 이미지에 굽지 않는다.
- 이미지·GIF·영상은 외부 자산으로 분리하며 안정적 ID, 해시, 대체 텍스트, 근거와 계보를 가진다.
- 실제 제품을 보존해야 하는 장면과 분위기 보조 생성이 가능한 장면을 구분한다.
- 모델 장면은 실제 인물 픽셀과 합성 성인 모델을 구분하고, 제품 상단·타공·귀·하단 등 주장에 필요한 부위를 프롬프트와 안전영역에 명시한다.
- 치수 장면은 A~E 값·시작점·끝점·검증 상태를 기록하고 숫자·치수선은 이미지가 아니라 HTML/SVG로 표현한다.
- 제품 형태를 변경할 위험이 있는 편집은 합성 또는 실사진 촬영으로 대체한다.
- GIF는 하나의 동작 또는 효용을 3~6초 내에 증명하도록 계획한다.
- 치수, 착용감, 방수·성능 등 실증이 필요한 장면은 실사진 또는 실촬영을 우선한다.
- 실패한 소재 또는 HTML 모듈만 수정하고 이력을 남긴다.
- 실제 제품 단독 원본이 없는 품질 프리뷰는 사용자가 범위를 명시적으로 허용한 `concept_only` 격리 경로에서만 만들 수 있다.
- `concept_only`의 생성 마스터와 장면은 `trusted_for_identity=false`이며 모든 화면에 연출 이미지임을 표시하고 실제 제품 근거·게시 패키지로 승격하지 않는다.
- GIF는 정적 대체 자산, 재생 정지와 `prefers-reduced-motion` 대응을 제공한다.

### FR-10. 소재 QA·통합 QA와 산출물 승격

- 소재 자동 검사는 파일·해시·형식·참조·출처 연결을 확인하고, 육안 검사는 제품 동일성·주장-증거·시각 품질을 확인한다.
- 통합 자동 검사는 계획·자산·HTML·CSS 해시, 반응형, 대체 텍스트와 참조 완결성을 확인한다.
- 통합 자동 검사는 모듈 순서, `data-claim-ids`와 `data-asset-id`의 연결, 주 피사체 95% 이상과 핵심 부위 100% 가시율을 실제 브라우저 좌표로 확인한다.
- 통합 육안 검사는 전체 구매 흐름, 브랜드 일관성, 모바일 가독성, 접근성, 광고 위험과 채널 적합성을 확인한다.
- 주장-이미지 연관성은 100점 척도에서 모듈별 80점 이상이어야 하고, 핵심 주장은 하나 이상의 직접 시각 모듈을 가져야 한다.
- 실패 시 원인이 발생한 단계로만 롤백한다.
- 소재 QA와 통합 QA가 모두 통과하지 않은 결과는 최종 패키지로 승격하지 않는다.
- 소재 QA와 통합 QA의 공통 필드·판정·실사 교체 순서는 [콘텐츠 생성·조립 QA 체크리스트](CONTENT-PRODUCTION-CHECKLIST.md)를 따른다.
- `concept_only`는 시각 일관성과 조립 품질을 `pass` 또는 `pass_with_limit`로 평가할 수 있지만 실제 제품 동일성은 항상 `blocked`로 남긴다.

### FR-11. 판매 신호 게이트와 2차 심화검증

- 1차 테스트 출시 결과에서 판매량·전환율·광고 효율·공헌이익·반품·문의·별점·후기를 수집한다.
- 현재 2차 진입 기준은 캠페인 시작 후 14일 이내 `판매수량 2개 이상 AND ROAS 400% 이상`이며 실제 값은 사용자가 제공한다.
- ROAS는 `광고 귀속 매출 ÷ 광고비 × 100`으로 계산하며 광고비 0원 또는 귀속 범위 미확인 상태는 통과시키지 않는다.
- 캠페인 ID별 기간·광고비·귀속 매출·판매수량·ROAS·판정과 사용자 보고 시점을 분리해 기록한다.
- SKU 승격에는 사용자가 지정한 판정 캠페인을 사용하며, 여러 캠페인의 합산에는 대상과 방식의 별도 승인이 필요하다.
- 2차 진입 기준과 판정 방식은 정책 버전으로 기록하고 변경 시 기존 결과를 덮어쓰지 않는다.
- 기준 통과 제품만 다수 샘플·심화 실험·검사서 확인과 확장 콘텐츠 투자 대상으로 승격한다.
- 2차 검증 결과는 제품 사실 원장, 상품기획, Source of Truth, 콘텐츠기획, 이미지+HTML과 QA를 갱신한다.
- 기준 미달 제품은 개선 후 재테스트, 제한 판매, 재소싱 또는 판매 중단으로 판정한다.

### FR-12. 브라우저 UI와 프로젝트 작업공간

- 로컬 React 대시보드는 프로젝트 목록, 현재 단계, 9단계 전체 흐름, 필수 입력, 승인 게이트와 실제 결과를 표시한다. 소싱 실행 중에는 `진행중입니다.`만 표시한다. 실행 실패뿐 아니라 실행 완료 후 차단·검증 미완료·pair 0건도 실제 실패 이유와 현재 실행의 도매꾹 샘플을 표시하고, 실제 pair가 있으면 도매꾹↔쿠팡 pair와 보고서를 표시한다.
- 화면 정보 순서는 `현재 할 일 → 필요한 입력 → Codex 실행·로그 → 결과 확인·승인 → 다음 단계`를 기본 흐름으로 사용하고, 현재 단계의 주행동을 파일·폴더 정보보다 먼저 노출한다.
- UI는 쿠팡의 공개 RDS 원칙을 참고하되 로고·다색 워드마크·비공개 토큰을 복제하지 않는 독립 커머스 색상 토큰과 `Commerce Flow` 식별자를 사용한다.
- 9단계 탐색은 선택 단계에 `aria-current="step"`을 제공하고 360px·800px에서 전체 페이지 가로 넘침 없이 현재 행동과 내부 가로 단계 탐색을 유지한다.
- 사용자는 UI에서 프로젝트를 생성하고 선택하며 단계 입력과 승인 상태를 실제 `project.json`에 저장할 수 있어야 한다.
- 새 프로젝트는 `commerce-project/projects/<project-id>/` 아래에 입력·소싱·상품기획·콘텐츠기획·자산·상세페이지·QA·피드백 폴더를 만든다.
- 프로젝트 ID는 3~64자의 소문자 영문·숫자·하이픈으로 제한하고 기존 프로젝트를 덮어쓰지 않는다.
- `reports/YYYY/YYYY-MM-DD/<run-name>/` 산출물은 복사하지 않고 프로젝트에 상대 경로로 연결한다.
- 프로젝트 작업창은 실제 프로젝트 파일과 연결 보고서를 VS Code형 탐색기로 표시하고 HTML·이미지·JSON·텍스트를 같은 화면에서 미리본다.
- 사용자는 PNG·JPG·GIF·WEBP 이미지를 탐색기와 이미지가 필요한 단계의 인라인 드롭존에 놓아 `folderMap.sourceAssets`에 저장할 수 있어야 한다. 단계 폼에는 이미지 폴더 경로 입력을 두지 않고 폴더 안 기존 이미지를 썸네일·확대 미리보기로 제공한다. 파일당 20MB, 이미지 MIME·시그니처, 안전한 파일명, 중복 덮어쓰기 금지를 서버에서 검사한다.
- 프로젝트·보고서 HTML 미리보기는 스크립트를 차단하는 CSP와 샌드박스 iframe을 사용한다.
- 기존 `detail-page/projects/`는 자동 이동·삭제하지 않고 읽기 전용 레거시 목록으로 노출한다.
- UI는 현재 단계에 맞는 전문 스킬 한 개를 `codex --ask-for-approval never --search exec --json --sandbox workspace-write --ephemeral`로 실행하고 진행 이벤트와 최종 응답을 읽기 전용 내장 콘솔에 표시한다. 설치된 전문 스킬과 Browser Use 도구는 읽되 샌드박스와 승인 정책은 명령행에서 고정한다.
- 소싱 자동 실행은 종료 코드 0만으로 완료되지 않는다. 신규 HTML·JSON 보고서가 `reports/`에 생성되고 `links.reportRuns`에 등록되어야 하며, UI는 실행 종료 뒤 프로젝트를 다시 읽고 보고서 링크를 열 수 있게 표시한다.
- 같은 프로젝트의 Codex 작업은 한 번에 하나만 실행하고 사용자가 해당 실행을 중지할 수 있어야 한다.
- 잠긴 미래 단계는 앞 단계 통과 전 실행하지 않는다. 단, UI 밖에서 상품·가격 선택을 끝낸 사용자가 상품기획의 `앞 단계 완료 확인`을 체크하면 앞 단계 완료·승인 레코드를 위조하지 않은 별도 확인 상태로 상품기획을 열고 기존 `coupang-product-planning`이 SKU·가격·공급처 근거를 다시 검증한다. 비대화형 실행에서 추가 승인이 필요한 작업은 권한을 자동 확대하지 않고 실패 상태를 표시한다.
- UI의 완료 체크는 전문 스킬의 테스트·QA 결과를 대체하지 않으며 외부 주문·게시·광고 행위를 실행하지 않는다.

## 6. 비기능 요구사항

- 추적성: 모든 주요 주장과 자산은 출처 또는 승인 기록을 가져야 한다.
- 재현성: 동일 입력과 정책 버전으로 평가 및 검증 스크립트를 다시 실행할 수 있어야 한다.
- 안전성: CAPTCHA 우회, 무단 로그인, 자동 주문·결제·상품 등록·광고 집행을 수행하지 않는다.
- 변경 격리: 제품별 프로젝트 산출물을 분리하고 다른 SKU의 사실·브랜드·자산을 재사용하지 않는다.
- 실패 복구: 전체 파이프라인이 아닌 가장 가까운 실패 게이트로 롤백할 수 있어야 한다.
- 확장성: 공급처, 판매 채널, 렌더링 방식은 공통 계약을 유지한 채 어댑터로 확장할 수 있어야 한다.
- 사용성: 사용자 승인이 필요한 지점과 현재 상태, 차단 사유, 다음 행동이 명확해야 한다.
- 프로젝트 탐색성: 한 상품의 상태·폴더·실제 파일·보고서·HTML·이미지 미리보기를 프로젝트 화면 하나에서 찾을 수 있어야 한다.
- 로컬성: 프로젝트 UI와 API는 `127.0.0.1`에서 동작하고 입력을 외부 서비스로 전송하지 않아야 한다.

## 7. 상태 모델과 주요 산출물

권장 상태 흐름은 다음과 같다.

```text
initialized
→ sourced
→ awaiting_user_selection
→ shortlist_approved
→ product_verified
→ researched
→ product_plan_approved
→ content_plan_approved
→ materials_qa_passed
→ assembled
→ integration_qa_passed
→ packaged
```

주요 산출물은 다음과 같다.

- 소싱: 평가 JSON, 가격 시나리오, `qualified-candidates.html`, 선택 대기 상태
- 제품 검증: 제품 사실 JSON, 사실 원장, 제품 불변 특성, 자산 계보
- 조사: 경쟁상품·리뷰 출처, 고객 불안 지도, 기획 원칙
- 상품기획: 고객·문제·포지셔닝·오퍼·주장 JSON과 사용자 승인 해시
- 콘텐츠기획: 모듈별 카피·근거·자산·QA 명세와 사용자 승인 해시
- 제작: 외부 이미지·GIF·영상 자산, 자산 manifest, 실사진 촬영 목록
- 조립: 편집 가능한 HTML·CSS와 패키지 manifest
- QA: 소재별 자동·육안 QA, 조립 후 통합 자동·육안 QA, 재생성 로그
- 판매 피드백: 캠페인별 사용자 성과 기록, 판정 정책 버전, 2차 진입 판정과 재승인 이력
- 프로젝트 UI: `project.json`, 단계별 폴더 지도, 보고서 링크, 현재 단계·차단·승인 상태

## 9. 수용 기준

플러그인의 초기 제품 목표는 아래 조건을 모두 만족할 때 수용한다.

1. 플러그인 매니페스트 검증을 통과하고 오케스트레이터·프로젝트 UI를 포함한 여덟 스킬을 정상 발견할 수 있다.
2. 소싱 실행이 기준 통과 후보 5개 이상 또는 명시적인 부족 사유를 포함한 보고서를 생성한다.
3. 모든 추천 후보에 공급처 및 시장 근거 URL과 두 개의 마진 시나리오가 존재한다.
4. 사용자 선택 전 상세페이지 단계가 시작되지 않는다.
5. 상세페이지 프로젝트가 제품 사실, 자산 계보, 시장 조사, 브랜드 승인 및 장별 근거를 보존한다.
6. 신규 프로젝트가 편집 가능한 HTML·CSS와 외부 시각 자산을 산출하거나 차단 사유를 명확히 보고한다.
7. 각 소재와 조립 결과가 각각 자동·육안 검수를 모두 통과하며, 실패 항목은 해당 단계로 롤백된다.
8. 근거 없는 광고 표현과 제품 외형 변형이 최종본에 포함되지 않는다.
9. 1차 미검증 가설은 판매 주장과 분리되고, 판매 신호를 통과한 제품의 2차 검증 결과가 재승인·재QA를 만료 또는 갱신한다.
10. 모든 콘텐츠 모듈이 스토리보드의 주장·근거·자산에 연결되고 순서·크롭·타이포·주장-이미지 연관성 수치형 AC를 통과한다.
11. 완성 HTML과 미수행 실험 계획이 별도 산출물로 제공되며 실제 실험 결과처럼 혼동되지 않는다.
12. 사용자는 브라우저에서 프로젝트를 생성·선택하고 현재 단계·필수 입력·승인·차단·폴더·보고서 링크를 확인할 수 있다.
13. 신규 프로젝트는 정해진 단계 폴더를 만들고 기존 `reports/`와 `detail-page/projects/`를 이동·복사·덮어쓰지 않는다.
14. 도매꾹과 쿠팡 소싱 브라우저는 사용자 Chrome과 분리된 로컬 headless 임시 세션에서 실행해 활성 창과 키보드 포커스를 가져가지 않는다. headless 실패 시 표시형 브라우저로 자동 전환하지 않고 차단 상태를 보고한다.

| 판매 신호 없이 심화검증 비용이 증가함 | 사용자 승인 판매 기준을 통과한 상품만 2차 승격 |
| UI 완료 체크를 실제 QA 통과로 오인 | 진행 메모와 전문 스킬 게이트를 분리하고 UI에 경계를 상시 표시 |
| 기존 작업 폴더 자동 이동으로 링크 손상 | 레거시는 읽기 전용 발견만 하고 명시적 연결 전 이동·삭제 금지 |

## 11. 출시 단계

- Phase 1: 여덟 스킬·React 프로젝트 대시보드 안정화, 분리 기획·시각 스토리보드·하이브리드 HTML·2층 QA 계약 강화
- Phase 2: 구현된 프로젝트 폴더·상태 위에 사용자 선택 후보의 전체 근거 자동 승격과 공통 증거 계약 도입
- Phase 3: 모션, 쿠팡 정적 렌더링, 채널 패키징 및 게시 전 QA 추가
- Phase 4: 판매 신호 게이트, 2차 심화검증·성과 피드백과 공급처·채널 어댑터 확장

## 13. 용어 정의

| 용어 | 정의 |
|---|---|
| 후보 | 아직 사용자가 선택하지 않은 소싱 대상 상품 |
| SHORTLIST | 수요·수익성·경쟁·운영 기준을 통과해 사용자 선택 대상으로 승격된 후보 |
| 가격안 | `entry`, `recommended`, `premium`, `reject` 중 근거와 함께 제시하는 판매가 전략 |
| 제품 사실 잠금 | 최종 제작에서 바뀌면 안 되는 제품 정보와 시각적 불변 특성을 확정하는 절차 |
| 증거 | 주장 또는 시각 표현을 뒷받침하는 사용자 확인, 공식 출처, 실제 이미지나 시장 관찰 |
| 게이트 | 필수 검증과 승인이 끝나야 다음 단계로 이동할 수 있는 품질 경계 |
| 승격 | 검증된 산출물을 다음 상태 또는 최종 출력 위치로 이동시키는 행위 |
| 1차 간이검증 | 최소 사실과 가능한 간이 실험으로 소구 가설과 제한된 주장 범위를 만드는 시장반응 준비 단계 |
| 2차 심화검증 | 판매 신호를 통과한 SKU에 다수 샘플·후기·검사서·심화 실험을 적용해 확장 가능성을 확인하는 단계 |
