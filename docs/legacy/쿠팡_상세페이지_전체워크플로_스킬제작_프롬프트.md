## 스킬의 핵심 운영 원칙

### 1. 제품 원본은 절대 기준이다

사용자가 처음 제공한 원본 상품 이미지를 제품 정체성의 유일한 source of truth로 취급해라. 도매 상세페이지, 경쟁상품, 레퍼런스 이미지는 상품 정보와 구성 아이디어를 얻는 자료일 뿐 제품 형태의 기준이 아니다.

원본 이미지마다 역할을 명시해라.

- `PRODUCT_TRUTH`: 실제 제품 형태·재질·색상·디테일 기준
- `PRODUCT_ANGLE`: 정면·측면·후면·밑창·펼침·착용 상태 등 각도 기준
- `REFERENCE_LAYOUT`: 상세페이지 구성과 분위기 참고
- `COMPETITOR`: 경쟁상품 분석용이며 제품 생성 참조 금지
- `REAL_PHOTO_TARGET`: 최종적으로 실사진으로 교체할 위치

인물이 포함된 원본은 인물의 얼굴, 체형, 의상 정체성을 참조하지 말고 제품과 제품이 닿는 최소 신체 영역만 참조해라.

### 2. 사실, 추론, 미확인을 분리한다

모든 상품 정보를 `product-truth-ledger.md`에 기록하고 각 항목을 다음 중 하나로 표시해라.

- `VERIFIED_SOURCE`: 도매 페이지, 공식 표기, 포장 또는 실물로 확인됨
- `VERIFIED_VISUAL`: 원본 이미지에서 직접 확인됨
- `USER_CONFIRMED`: 사용자가 명시적으로 확인함
- `INFERRED`: 합리적 추론이지만 상세페이지 카피로 사용 금지
- `UNVERIFIED`: 확인되지 않음
- `PROHIBITED_CLAIM`: 근거 없이 사용하면 안 되는 표현

다음 필드를 기본으로 확인해라.

- 상품명과 검색형 판매명
- 브랜드명
- 카테고리와 사용 목적
- 구성품과 판매 단위
- 소재와 혼용률
- 색상과 실제 투명도
- 모든 사이즈 옵션
- 실측 길이·폭·높이·둘레·무게
- 제조국, 제조자, 수입자
- 사용 방법
- 관리·세척·건조·보관 방법
- 주의사항과 부적합 사용 환경
- 인증·시험·수치
- 사용 가능한 신발·대상·환경
- 사용하기 어려운 신발·대상·환경

미확인 정보는 임의로 채우지 마라. 중요한 정보가 빠졌으면 먼저 안전하게 조사하고, 그래도 확인되지 않으면 `촬영·공급사 확인 필요`로 표시한 뒤 해당 문구와 장면을 최종 생성에서 제외해라.

### 3. 경쟁상품 리뷰는 광고 카피가 아니라 불안 해소에 사용한다

사용자가 도매 사이트와 쿠팡 조사를 요청하면 공개 웹 검색을 먼저 사용하고, 검색 결과만으로 리뷰·상세 구성 확인이 부족할 때 공개 브라우저 탐색 도구를 사용해라. `browser-use` 스킬이나 사용 가능한 동등한 공개 브라우저 제어 스킬이 있으면 그 지침을 먼저 읽고 사용한다. 인증이 필요한 정보는 브라우저로 우회하지 마라.

가능하면 다음 세 종류를 각각 조사해라.

1. 동일 제품 또는 동일 모델
2. 같은 구조의 직접 경쟁상품
3. 문제를 다른 방식으로 해결하는 대체상품

각 상품에서 다음을 수집해라.

- 상품명과 링크
- 가격대와 옵션 구조는 조사 시점 참고값으로만 기록
- 리뷰 수와 평점은 근거 링크와 조사일을 함께 기록
- 반복적으로 나타나는 긍정 요인
- 반복적으로 나타나는 불만 요인
- 반품·문의로 이어질 가능성이 높은 혼동
- 경쟁 상세페이지의 전개 순서
- 경쟁사가 강하게 주장하지만 근거가 약한 표현
- 경쟁사가 충분히 설명하지 않는 정보

리뷰 문장을 길게 복사하지 말고 요약해라. 한두 리뷰를 전체 시장 의견으로 일반화하지 마라. 같은 불만이 여러 상품·리뷰에서 반복되는지 확인하고 빈도는 `높음/중간/낮음/확인 부족`으로 표시해라.

`competitor-pain-map.md`에는 다음 표를 만든다.

| 소비자 불편·불안 | 근거 | 반복 수준 | 구매에 미치는 영향 | 우리 제품이 실제로 답할 수 있는가 | 상세페이지 대응 | 필요한 증거 |

우리 제품이 실제로 해결하지 못하는 경쟁사 약점은 소구점으로 사용하지 마라. 예를 들어 미끄럼 시험이 없으면 밑창 돌기를 보여줄 수는 있지만 `절대 미끄럽지 않음`을 주장하지 마라. 내구성 시험이 없으면 `안 찢어짐` 대신 `입구를 과도하게 늘리는 동작을 줄이는 구조`처럼 구조적 사실만 말해라.

### 4. 디자인 전에 구매 논리를 확정한다

조사 후 바로 이미지를 생성하지 마라. 먼저 다음 네 가지를 사용자와 확정해라.

1. 핵심 타겟 한 그룹
2. 대표 문제 한 가지
3. 핵심 소구점 세 가지
4. 사용하면 안 되는 표현

기본 구매 설득 흐름은 다음을 따른다.

`문제 공감 → 경쟁상품 사용 불안 → 우리 제품의 해결 방식 → 구조 증거 → 실제 사용 장면 → 사이즈 선택 → 적합·부적합 대상 → 관리·주의사항 → 사용 상황 → 최종 요약`

기본은 10장으로 구성하되 제품 특성상 안전·사이즈 정보가 많으면 한 장의 광고 밀도를 높이지 말고 사용자에게 장수 확대 여부를 제안해라. 사용자가 10장을 고정하면 정보 카드와 사용 장면을 조합해 10장 안에 수용한다.

### 5. UI 디자인 시스템을 이미지 생성 전에 잠근다

페이지별로 제각각 생성하지 마라. 먼저 `ui-guide.md`를 만들고 사용자 승인 또는 명시적 확정을 받은 뒤 모든 장에 적용해라.

UI 가이드에 다음을 포함해라.

- 캔버스: 기본 800×2400px 세로형, 사용자가 다른 규격을 주면 그 규격 우선
- 안전 여백: 좌우 최소 64px, 상하 주요 카피 영역 최소 96px
- 콘텐츠 그리드: 8px 기반 간격 체계
- 정보 밀도: 한 장당 핵심 메시지 1개
- 헤드라인: 최대 2줄
- 보조 문구: 최대 2개
- 모바일 최소 글자 크기 기준
- 한글 폰트 계열과 굵기
- 브랜드 컬러 3–5개와 각 역할
- 배경색과 대비 기준
- 카드 모서리 반경
- 카드 내부 여백
- 아이콘 스타일
- 제품 그림자와 접지 방식
- 사진 크롭 규칙
- 물방울·빛·장식의 사용량
- 비교 카드, 디테일 확대, 단계 카드, 주의 카드, 사이즈표 컴포넌트
- 페이지 사이의 반복 요소와 변화 요소

가능하면 UI 토큰을 표로 만든다.

| 토큰 | 값 | 용도 |

각 장의 프롬프트는 UI 가이드의 토큰을 재선언하지 말고 참조하도록 구성하되, 이미지 생성 도구에 넣는 최종 프롬프트에는 필요한 핵심 토큰을 짧게 반복해 일관성을 유지한다.

### 6. 한글과 UI는 다단계 이미지 생성·편집으로 통합한다

단순히 완성 이미지 위에 폰트를 얹는 결정론적 텍스트 합성을 기본값으로 사용하지 마라. 글자는 정확해도 배경의 빛, 질감, 카드, 물방울, 강조 효과와 분리되어 보여 상업 상세페이지의 완성도가 떨어질 수 있다.

기본 제작 방식을 다음의 `Base → Design Edit → Local Repair` 3단계로 설정해라.

#### Stage 1 — Base Image Generation

1. 제품 원본 보존 방식에 따라 제품·배경·주요 레이아웃을 먼저 생성하거나 합성한다.
2. 헤드라인, 보조 문구, 배지와 카드가 들어갈 공간을 확보하되 아직 글자를 넣지 않는다.
3. 이 결과를 다음 이미지 편집 단계의 정확한 edit target으로 사용한다.
4. 제품이 크게 보이는 장은 생성된 제품이 아니라 원본 제품 픽셀을 합성한 베이스를 우선한다.

#### Stage 2 — Integrated Design Edit

1. Stage 1 이미지를 다시 이미지 생성·편집 도구에 입력한다.
2. 제품, 손, 신체, 배경의 핵심 구도는 변경하지 않고 UI 카드, 타이포그래피, 강조선, 물방울, 배지와 한글 카피만 디자인적으로 통합한다.
3. 글자가 단순 오버레이처럼 떠 보이지 않도록 배경 조도, 카드 재질, 그림자, 글자 주변 여백과 강조 효과를 함께 조정한다.
4. 제품 영역에는 편집이 침범하지 않게 하고 가능한 경우 마스크 또는 국소 편집 영역을 사용한다.
5. 마스크 기능이 없으면 프롬프트에 `제품 픽셀과 외곽선은 변경하지 말고 상단·카드·텍스트 영역만 수정`을 반복한다.

#### Stage 3 — Local Text Repair

1. 한글 오탈자나 깨진 글자가 있으면 전체 페이지를 처음부터 다시 만들지 않는다.
2. 직전 완성 이미지를 다시 edit target으로 사용한다.
3. 틀린 텍스트 카드 또는 헤드라인 영역만 국소 재생성한다.
4. 제품·배경·인물·나머지 UI는 픽셀 수준으로 유지하도록 지시한다.
5. 같은 텍스트가 두 번 실패하면 문구를 더 짧게 줄이되 의미 변경은 사용자 승인 후 진행한다.
6. 이미지 편집으로도 정확한 한글을 얻지 못한 경우에만 마지막 폴백으로 정교한 텍스트 합성을 사용하고, 이 사실을 QA 보고서에 표시한다.

각 장마다 다음 세 파일 또는 버전을 구분해 관리해라.

- `page-XX-base`: 제품·배경·레이아웃 베이스
- `page-XX-designed`: UI와 한글이 통합된 디자인 버전
- `page-XX-final`: 국소 교정과 QA가 끝난 최종 버전

이미지 생성·편집으로 한글을 통합할 때 다음을 지켜라.

- 지정 문구만 사용
- 헤드라인 1개, 보조 문구 최대 2개
- 제품 위에 글자 인쇄 금지
- 원근·굴곡·물결 변형 금지
- 깨진 글자가 있으면 해당 텍스트 영역만 다시 편집
- 의미 없는 영어와 숫자 금지

각 Stage 2 프롬프트에 다음 편집 잠금 문구를 포함해라.

`입력 이미지를 정확한 편집 기준으로 사용한다. 제품의 외곽선, 형태, 색상, 투명도, 밴드·단추·밑창, 신발, 손과 발, 카메라 구도는 변경하지 않는다. 지정된 텍스트 안전영역과 UI 카드만 편집한다. 지정한 한글을 글자 단위로 정확하게 렌더링하고 다른 문자는 만들지 않는다.`

### 7. 장마다 자산 전략을 먼저 판정한다

이미지 생성 전에 `asset-strategy.md`를 만들고 각 장의 각 제품 요소를 다음 모드 중 하나로 분류해라.

- `PRESERVE`: 원본 픽셀을 그대로 사용
- `COMPOSITE`: 원본 제품을 분리해 새 배경에 합성
- `EDIT_BACKGROUND_ONLY`: 제품은 고정하고 배경·날씨·조명만 수정
- `GENERATE_SUPPORT`: 배경, 소품, 아이콘, 일반 상황만 생성
- `GENERATE_PRODUCT_ALLOWED`: 원본과 달라도 문제가 없는 낮은 위험 제품 표현에만 제한적으로 허용
- `REAL_PHOTO_REQUIRED`: AI 생성 금지, 실사 촬영 후 교체
- `GIF_REQUIRED`: 정지 이미지보다 동작 증거가 중요한 영역

표에는 다음을 포함해라.

| 장 | 요소 | 모드 | 원본 파일 | 허용 변경 | 금지 변경 | 제품 왜곡 위험 | 최종 교체 계획 |

다음 조건에서는 `COMPOSITE` 또는 `PRESERVE`를 강제한다.

- 제품이 화면의 25% 이상을 차지하는 히어로 장면
- 제품의 펼침 구조를 설명하는 장면
- 밴드·단추·지퍼·라벨 등 구조적 디테일 장면
- 밑창 패턴 장면
- 화이트·블랙 등 옵션 비교
- 사이즈 비교
- 첫 장과 마지막 구매 결정 장
- 원본과 다르면 고객 오인이 생기는 모든 장

다음 조건에서는 `REAL_PHOTO_REQUIRED`를 강제한다.

- 사이즈별 실측
- 구성품과 실제 포장
- 제품 두께와 무게
- 실제 착용 순서
- 세척과 건조 방법
- 밑창 전체
- 스냅·밴드·마감의 근접 디테일
- 테스트 수치나 성능을 증명하는 장면

원본에 존재하지 않는 접힌 각도, 제품 후면, 내부 구조, 밑창 각도를 AI가 상상해 만들지 마라. 필요한 각도의 원본이 없으면 촬영 목록에 넣고 임시 레이아웃에는 빈 교체 프레임을 사용해라.

### 8. 제품 정체성 잠금 규칙을 만든다

제품마다 `product identity lock`을 작성한다. 최소한 다음을 기록해라.

- 전체 실루엣과 비율
- 정확한 색상과 투명도
- 소재 질감과 두께 인상
- 주요 부품의 개수
- 각 부품의 상대 위치
- 앞·뒤·좌·우 비대칭 요소
- 밑창 패턴
- 주름, 라벨, 봉제, 밴딩, 버클, 단추 등 세부 특징
- 착용 시 변형되는 부분과 고정되는 부분
- 생성 모델이 자주 틀리는 요소

각 이미지 생성 또는 편집 프롬프트에 `절대 변경하지 않을 항목`으로 짧게 포함해라. 생성 후 다음을 체크해 하나라도 실패하면 승인하지 마라.

- 부품 수 일치
- 부품 위치 일치
- 좌우 제품 동일 모델
- 색상·투명도 일치
- 앞코·뒤꿈치·높이 비율 일치
- 밑창 패턴 일치
- 임의 로고·라벨·지퍼 없음
- 신발과 커버가 융합되지 않음

### 9. 상세페이지 10장 산출물

각 장에 대해 다음 정보를 표로 먼저 작성해라.

| 장 | 역할 | 해결할 고객 불안 | 헤드라인 | 보조 문구 | 제품 증거 | 레이아웃 | 자산 전략 | 실사진 교체 | GIF 후보 | 주의 표현 |

10장의 기본 역할은 다음과 같이 설계하되 제품에 맞게 조정한다.

1. 대표 문제 후킹
2. 경쟁 리뷰에서 반복된 불편·구매 불안
3. 제품 메인 소개와 핵심 해결 방식
4. 일반적인 방식과 사용 구조 비교
5. 핵심 디테일과 구조 증거
6. 실제 사용 상황과 기능 범위
7. 사이즈·옵션 선택 안내
8. 적합한 대상과 주의할 대상
9. 착용법·관리법·주의사항
10. 사용 상황과 핵심 장점 요약, 구매 결정

제품 정보가 부족하면 7–9장을 임의로 생성하지 말고 `실물 확인 후 완성` 상태로 표시한다.

각 장의 이미지 생성 프롬프트는 다음 구조로 만든다.

- 사용 목적
- 입력 이미지별 역할
- 제품 identity lock
- 장면과 배경
- 제품 배치와 크롭
- UI 컴포넌트
- 텍스트 영역 위치
- 정확한 한글 문구
- 허용 변경
- 금지 변경
- 네거티브 프롬프트
- 한글 실패 시 직전 이미지 기반 국소 재편집 프롬프트
- 제품 영역 편집 잠금 문구

### 10. 생성 순서와 승인 게이트

한 번에 10장을 무검수로 생성하지 마라. 다음 게이트를 사용한다.

#### Gate A — 조사 승인

- 제품 사실표 완성
- 경쟁 불편 지도 완성
- 미확인 정보와 추가 질문 표시

#### Gate B — 전략 승인

- 타겟, 대표 문제, 핵심 소구점 3개 확정
- 금지 표현 확정
- 10장 구매 흐름 확정

#### Gate C — UI 승인

- UI 토큰과 핵심 컴포넌트 확정
- 한 장의 스타일 프레임 또는 와이어프레임 제작

#### Gate D — 제품 보존 시험

- 제품이 크게 보이는 1장 또는 3장을 먼저 제작
- 원본과 비교해 구조·색상·투명도·부품 수 검수
- 실패하면 생성 방식에서 합성 방식으로 전환

#### Gate E — 10장 제작

- 승인된 방식으로 각 장의 Base Image를 제작
- Base Image를 다시 편집 입력으로 사용해 UI·한글이 통합된 Designed Image를 제작
- 오탈자는 Designed Image의 해당 영역만 국소 재편집
- 제품 생성 위험이 높은 장은 원본 합성 사용

#### Gate F — 정보·시각 QA

- 한글, 수치, 제품 구조, 사이즈, 관리법, 주의사항 검수
- 10장 접촉 시트로 전체 톤과 반복 이미지 확인

사용자가 “한 번에 전부 생성해줘”라고 요청하면 중간 승인 질문을 생략할 수 있지만, 각 게이트의 내부 검수는 생략하지 마라. 치명적인 정보 부족이나 제품 왜곡이 있으면 생성을 멈추고 보고해라.

### 11. 실사진 교체 계획을 필수 산출물로 만든다

AI 상세페이지 시안이 완성된 뒤 `photo-shot-list.md`를 반드시 작성한다. 다음 표를 포함해라.

| 컷 ID | 교체할 장·영역 | 촬영 목적 | 필수 제품 상태 | 구도·렌즈 | 배경·조명 | 손·인물 필요 여부 | 측정 도구 | 교체 방법 | 우선순위 |

기본 촬영 목록:

- 제품 정면·측면·후면
- 제품 펼침 상태
- 밴드·단추·라벨 등 핵심 디테일
- 밑창 전체
- 색상 옵션 비교
- 모든 사이즈 비교
- 자를 이용한 길이·폭·높이 실측
- 실제 구성품과 포장
- 착용 전·중·후 단계
- 사용 후 세척·건조·보관
- 실제 사용 환경

각 촬영 컷이 어느 생성 이미지의 어떤 영역을 교체하는지 명확히 연결해라. 단순한 촬영 아이디어 목록으로 끝내지 마라.

### 12. GIF·모션 소구점 계획을 필수 산출물로 만든다

정지 이미지로 이해하기 어려운 소구점은 `gif-plan.md`에 계획한다. GIF는 장식이 아니라 제품 구조와 사용 편의성을 증명하는 용도로만 사용한다.

GIF 후보 선정 기준:

- 동작 전후 차이가 3–6초 안에 이해되는가
- 제품의 정체성을 원본 영상 또는 실사로 유지할 수 있는가
- 반복 재생해도 시작과 끝이 자연스러운가
- 정지 이미지보다 구매 불안을 더 잘 해소하는가
- 과장된 성능을 암시하지 않는가

기본 GIF 후보:

1. 밴드 열기 → 신발 넣기 → 똑딱 잠그기
2. 일반 입구형의 당기는 동작 vs 오픈형의 펼치는 동작
3. 제품 표면에 일상적인 빗물이 흐르는 장면
4. 사용 후 물로 헹구고 그늘 건조하는 장면
5. 제품을 무리 없이 접어 가방에 넣는 장면
6. 밑창 앞·뒤 돌기 디테일을 보여주는 짧은 회전

각 GIF에 다음을 작성해라.

| GIF ID | 해결할 불안 | 길이 | 첫 프레임 | 동작 구간 | 마지막 프레임 | 루프 방식 | 촬영 우선/AI 가능 | 제품 고정 요소 | 금지 동작 | 삽입 위치 |

실제 제품 동작이 중요한 GIF는 AI 영상 생성보다 실사 촬영을 우선한다. AI 영상은 배경의 빗물, 물방울, 부드러운 카메라 움직임 등 제품 형태를 변경하지 않는 보조 효과에만 사용한다. 라벨, 단추, 밴드, 텍스트가 프레임마다 변하면 실패로 판정한다.

플랫폼의 GIF 크기·용량·재생 제한을 사용자가 제공하지 않았다면 수치를 추측하지 말고, 최종 인코딩 전에 최신 업로드 요구사항을 확인하거나 사용자에게 요청해라.

### 13. QA 체크리스트

`qa-report.md`에서 각 장을 PASS, FIX, BLOCKED 중 하나로 판정해라.

#### 제품 일관성

- 원본과 실루엣이 같은가
- 색상과 투명도가 같은가
- 부품 수와 위치가 같은가
- 좌우 제품이 같은 모델인가
- 밑창과 라벨이 원본과 같은가
- 제품이 신발 자체나 일반 장화로 바뀌지 않았는가

#### 정보 정확성

- 모든 수치에 출처가 있는가
- 사이즈표가 옵션 정보와 일치하는가
- 소재·구성·제조국이 확인되었는가
- 관리법이 공급사 또는 실물 기준인가
- 리뷰·판매량·가격을 임의로 만들지 않았는가

#### 광고 안전성

- `100%`, `완벽`, `절대`, `최고`, `1위` 등 근거 없는 절대 표현이 없는가
- 시험 없는 방수·미끄럼·내구성 수치가 없는가
- 신체 효능이나 치료 표현이 없는가
- 경쟁사를 비방하거나 특정 브랜드를 복제하지 않았는가

#### UI·텍스트

- 캔버스 규격이 맞는가
- 안전 여백을 지켰는가
- 한 장에 메시지가 하나인가
- 모바일에서 글자가 읽히는가
- 한글 오탈자와 깨진 문자가 없는가
- 페이지 번호, 쿠팡 로고, 앱 UI가 없는가
- 10장의 컬러·카드·그림자·폰트가 일관적인가

#### 구매 설득

- 경쟁 리뷰의 핵심 불편이 앞부분에 반영되었는가
- 우리 제품이 실제로 해결할 수 있는 문제만 주장하는가
- 사이즈, 적합 대상, 관리, 주의사항이 포함되었는가
- 마지막 장까지 같은 메시지만 반복하지 않는가

### 14. 스크립트 요구사항

반복 검증을 위해 다음 스크립트를 구현하고 실제로 실행 테스트해라.

#### validate_project_manifest.py

- 프로젝트 매니페스트 필수 필드 검증
- 원본 이미지 역할 누락 확인
- 미확인 수치가 최종 카피에 사용되었는지 표시
- 페이지별 자산 전략과 실사진 교체 계획 누락 확인

#### audit_detail_images.py

- 이미지 수 확인
- 이미지 규격과 종횡비 확인
- 파일명 중복 확인
- 해시를 이용한 완전 중복 이미지 확인
- 각 장 QA 결과를 요약
- OCR은 신뢰 가능한 도구가 있을 때만 선택적으로 사용하고, 없으면 수동 텍스트 QA 항목을 남긴다.

#### build_contact_sheet.py

- 10장 미리보기를 순서대로 축소 배치한 접촉 시트 생성
- 각 장의 실제 이미지에는 페이지 번호를 넣지 않되, QA용 접촉 시트 바깥 라벨에는 파일명 표시 가능
- 제품 반복, 색상 불일치, UI 흐름을 한눈에 검토할 수 있게 한다.

모든 스크립트는 도움말과 명확한 오류 메시지를 제공하고, 샘플 프로젝트에 대해 실행 테스트해라.

### 15. 최종 산출물

스킬이 실제 작업에서 다음 산출물을 만들도록 해라.

1. 조사 출처와 조사일

## 전방 테스트 시나리오

스킬을 만든 뒤 최소 다음 시나리오로 전방 테스트해라.

### 테스트 A — 원본 구조가 복잡한 방수 신발커버

- 원본에 두 개의 스냅 밴드와 특수 밑창이 있음
- 경쟁 리뷰에는 찢어짐, 사이즈 실패, 미끄러움, 관리 불편이 있음
- 제품의 실측과 공식 관리법 일부가 누락됨
- 기대 행동: 미확인 정보를 만들지 않고 촬영·확인 목록으로 넘기며, 제품 히어로 장면은 생성보다 원본 합성을 선택함

### 테스트 B — 단순 섬유 제품

- 원본 색상과 라벨 보존이 중요함
- 상세 치수와 세탁법은 확인됨
- 기대 행동: 확인된 정보만 활용하고 UI 가이드를 먼저 만든 뒤 10장을 일관되게 생성함

### 테스트 C — 레퍼런스가 경쟁 제품과 혼재

- 사용자가 원본, 도매 이미지, 경쟁사 이미지, 분위기 레퍼런스를 함께 제공함
- 기대 행동: 이미지 역할을 분류하고 경쟁사 제품의 구조가 우리 제품에 섞이지 않게 함

전방 테스트에서는 결과가 다음을 만족하는지 확인해라.

- 제품 원본이 변형되지 않음
- 소비자 불편이 10장 초반에 반영됨
- 사이즈·관리·주의 정보가 누락되지 않음
- UI가 장마다 바뀌지 않음
- 실사진 교체와 GIF 계획이 구체적으로 연결됨
- 한글과 UI를 베이스 이미지 위의 2차 이미지 편집으로 통합하고 오탈자는 국소 재편집함

실패하면 원인을 SKILL.md 또는 reference·script·asset 템플릿에 반영하고 다시 검증해라.

완료 후 스킬이 무엇을 하는지, 포함한 핵심 파일, 검증한 내용, 사용자가 실제로 입력할 예시 요청을 알려줘.
