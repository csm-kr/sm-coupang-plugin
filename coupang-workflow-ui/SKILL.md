# 쿠팡 워크플로 UI

## 실행

1. `scripts/serve_workflow_ui.py --check --no-open`으로 패키지 자산을 검사한다.
2. 저장소 루트에서 `scripts/serve_workflow_ui.py --workspace .`를 실행해 로컬 서버와 기본 브라우저를 연다.
3. 대시보드에서 프로젝트를 선택하거나 새 프로젝트를 만든다.
4. 현재 단계의 필수 입력만 확인하고 `Codex 작업 시작`을 눌러 전문 스킬 한 개만 자동 호출한다. 새 프로젝트의 소싱 입력은 카테고리 `전체`, 도매꾹 개당 공급가 상한 `5,000원`, 최소 가격 배수 `3배`로 시작하며 사용자가 수정할 수 있다. 시작 기준은 `도매꾹 Best 고배수 pair 탐색`으로 고정한다.
5. `Codex 실행 콘솔`에서 JSONL 진행 이벤트를 확인하고 필요하면 실행을 중지한다. 소싱 결과 영역은 실행 중 `진행중입니다.`만 표시하고 완료 전 체크리스트나 내부 차단 코드를 노출하지 않는다.
6. 소싱 실행 자체가 실패했거나 실행은 끝났지만 보고서가 차단·검증 미완료·pair 0건이면 실제 실패 이유와 이번 실행 보고서의 도매꾹 확인 샘플을 상품명·원문 링크·카테고리·순위·확인 단가로 표시한다. 이전 실행 보고서는 실패 샘플로 섞지 않는다.
7. 소싱 완료 뒤 같은 화면의 `실제 결과`에서 도매꾹 상품·개당 원가와 쿠팡 판매상품·현재가, 가격 배수, 리뷰·만족 라벨을 pair로 표시한다. 낮은 가격의 다른 등록은 기준을 충족한 pair의 탈락 근거로 표시하지 않는다. 전체 소싱 검증을 통과한 `SHORTLIST` 한 건과 판매가를 확정하면 같은 동작에서 승인 상태를 저장하고 상품기획으로 이동한다.
8. 앞 단계 작업을 UI 밖에서 이미 끝낸 경우 상품기획의 `앞 단계 완료 확인`을 사용자가 체크해 상품기획 입력과 기존 `coupang-product-planning` 실행을 연다. 이 확인은 근거 통과를 대신하지 않으며 상품기획 스킬이 SKU·가격·공급처 근거를 다시 확인한다.
9. 상품기획은 공급처 상세 URL 하나로 조사를 시작할 수 있고 사이즈·구성·소재·관리법을 각각 필수 입력한다. 경쟁사 별점 1~3점 저평점 리뷰는 사용자에게 경로나 파일을 요구하지 않고 `coupang-product-planning`이 Browser Harness로 직접 조사한다. 조사 결과와 소구점 후보를 먼저 보여준 뒤 사용자가 판매가·묶음·판매 옵션·소구점을 고르는 오퍼 결정 게이트를 통과한다.
10. 이미지가 필요한 단계는 경로 입력 대신 단계 안 드롭존을 사용한다. `PROJECT EXPLORER`와 단계 자산 패널에서 프로젝트 폴더의 기존 HTML·이미지·JSON·텍스트를 미리보고, 실제 제품 이미지는 드래그앤드롭해 프로젝트 `40-assets/source`에 저장한다.
11. 전문 스킬의 근거·테스트·QA 결과를 확인한 뒤 완료 체크와 승인 게이트를 갱신한다.

소싱 실행은 과정 설명만 반환하지 않는다. 실제 조사 제품 또는 접근·근거 차단 사유가 담긴 HTML·JSON 보고서를 만들고, 설정 배수 이상인 도매꾹↔쿠팡 pair, 두 URL, 판매 근거와 가격 배수를 표시한 뒤 프로젝트 `links.reportRuns`에 연결한다.

현재 터미널을 점유하면 실행 명령을 별도 백그라운드 프로세스로 시작하되 서버 창은 숨긴다. 기본 포트가 사용 중이면 `--port 4174`처럼 다른 포트를 지정한다.

## 프로젝트 관리

- 새 프로젝트는 `<workspace>/commerce-project/projects/<project-id>/`에 단계별 폴더와 `project.json`을 만든다.
- `reports/`의 실행 보고서는 복사하지 않고 프로젝트의 `links.reportRuns`에 경로만 등록한다.
- 기존 `detail-page/projects/`는 자동 이동하지 않고 UI의 `기존 작업`에서 읽기 전용으로 보여준다.
- UI 저장은 로컬 API를 통해 원자적으로 `project.json`을 갱신한다. API가 연결되지 않으면 파일 상태를 갱신했다고 말하지 않는다.
- 후보 판단은 연결된 소싱 JSON을 UI 안에서 읽는다. `PRICE_REVIEW_BLOCKED`, `HIGH_MARKUP_DISCOVERY`, 조건부·탈락 후보는 확인할 수 있지만 `SHORTLIST`가 아니면 상품기획 단계로 넘기지 않는다.
- 사용자가 UI 밖에서 상품·가격 선택을 완료했다고 명시적으로 확인한 경우에만 상품기획을 직접 진입할 수 있다. 앞 단계 레코드를 완료로 위조하지 않고 사용자 확인 시각을 별도 보존하며, 상품기획 스킬은 앞 단계 근거를 재검증한다.
- 프로젝트 탐색기와 단계 자산 패널은 실제 프로젝트 파일과 연결 보고서를 읽고 HTML·이미지·JSON·텍스트를 인라인 미리보기한다. 이미지 입력 폼에 폴더 경로를 받지 않는다. 업로드 이미지는 PNG·JPG·GIF·WEBP, 파일당 20MB 이하만 허용하고 기존 파일을 덮어쓰지 않는다.
- Codex 실행은 저장소 루트에서 `codex --ask-for-approval never --search exec --json --sandbox workspace-write --ephemeral`로 시작하고 프롬프트는 표준입력으로 전달한다. 설치된 커머스 플러그인·Browser Use 도구를 사용하도록 사용자 구성을 읽되, 샌드박스와 승인 정책은 명령행에서 고정하고 세션은 저장하지 않는다.
- 같은 프로젝트에서는 Codex 실행을 한 번에 하나만 허용하고, 중지 시 서버가 시작한 해당 Codex 프로세스 트리만 종료하며 실행 이벤트는 서버 메모리에만 보관한다.
- 소싱 실행은 종료 코드가 0이어도 기존 링크에 없던 HTML·JSON 보고서 한 쌍이 `reports/` 아래 생성되고 `links.reportRuns`에 등록되지 않으면 실패다. 완료 뒤 프로젝트를 다시 읽어 새 보고서 링크를 즉시 표시한다.

## 안전 경계

- UI 체크만으로 실제 단계 산출물이 통과했다고 간주하지 않는다.
- 내장 콘솔은 임의 셸 명령 입력을 받지 않는 읽기 전용 Codex 실행 로그다.
- 자동 실행은 `workspace-write`로 제한한다. `danger-full-access`나 `--dangerously-bypass-approvals-and-sandbox`를 추가하지 않는다.
- 비대화형 실행은 `--ask-for-approval never`로 승인 팝업을 만들지 않으며, 샌드박스 안에서 허용되지 않는 작업은 즉시 실패 이벤트로 표시하고 권한을 자동 확대하지 않는다.
- 상품·가격, 상품기획, 콘텐츠기획, 최종 게시 승인은 사용자의 명시적 체크가 있어야 한다.
- 앞 단계가 완료되지 않은 미래 단계는 실행하지 않는다. 단, 사용자가 UI 밖에서 상품·가격 선택 완료를 명시적으로 확인한 상품기획 진입은 별도 사용자 확인 상태로 허용하고 전문 스킬이 근거를 다시 검증한다.
- 공급처 연락·발주·결제, 상품 등록, 광고 집행을 UI에서 자동 실행하지 않는다.
- 프로젝트 상태에 비밀번호·토큰·개인정보를 넣지 않는다.
- 드래그앤드롭 업로드는 프로젝트 `folderMap.sourceAssets` 밖에 쓰지 않으며 경로가 포함된 파일명, 형식 불일치, 중복 파일은 차단한다.
- `계획` 또는 `부분 지원` 단계의 구현 범위를 과장하지 않는다.

상태 JSON 필드와 단계 매핑을 수정할 때는 [워크플로 UI 계약](references/workflow-ui-contract.md)을 먼저 읽는다.
