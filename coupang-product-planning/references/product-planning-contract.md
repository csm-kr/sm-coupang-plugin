# 제품기획 산출물 계약

## 필수 산출물

- `product-truth-ledger.md`
- `product-facts.json`
- `competitor-pain-map.md`
- `claim-evidence-matrix.json`
- `planning/product-plan.json`
- `approvals/product-plan-approval.json`
- `experiment-plan.json`

## 입력과 조사 책임

- 공급처 상세 URL 하나만으로 조사를 시작한다. 제품기획 검색의 최소 입력에 색상·사이즈·구성·소재·관리법이나 최종 판매가를 포함하지 않는다.
- Codex는 Browser Harness로 공급처의 상세 본문·옵션·상품정보·공개 후기·문의를 확인해 색상·사이즈·구성·소재·관리법과 단가·MOQ·구매 단위·배송 조건을 조사한다. `product-truth-ledger.md`에 URL, 조사시각, 근거 위치, 원문 값, 정규화 값과 `확인 사실 / 공급처 주장 / 충돌 / 미확인 / 접근 차단` 상태를 기록한다.
- 상세 이미지 안의 치수·표는 원본 이미지를 실제 크기로 확인한다. 공급처 옵션·설명은 실수령 SKU 확정 근거가 아니므로 실제 사진·라벨·사용자 선택과 대조하고, 충돌하면 공급처 정보와 실물 정보를 모두 보존한다.
- Codex는 Browser Harness로 공개된 쿠팡 경쟁상품 5개 이상과 접근 가능한 별점 1~3점 리뷰를 조사한다. `competitor-pain-map.md`에는 경쟁상품 URL, 리뷰 URL 또는 공개 페이지 근거, 조사시각, 별점 범위, 관찰 표본 수와 반복 불만을 기록한다.
- 로그인·CAPTCHA·접근 통제를 우회하지 않는다. 공개 근거를 확보하지 못하면 불만을 추정하지 않고 차단 상태와 재개 조건을 기록한다.
- 검색 작업마다 사용자 브라우저와 분리된 임시 세션을 사용하고 `finally` 정리로 작업이 연 탭과 소유 브라우저 프로세스를 닫는다. 종료 뒤 작업 URL 탭과 소유 프로세스가 0건인지 확인한다.
- 사용자 앞 단계 완료 확인이나 기존 가격안은 진입 근거일 뿐 SKU·오퍼·공급처 검증을 대체하지 않는다.

## 검색 후 사용자 결정

- 조사 결과를 사용자에게 먼저 보고 확인 사실, 경쟁사 반복 불만, 해결 가능성, 미확인 사실과 가격·구성 제약을 한 번에 제시한다.
- **오퍼 결정 게이트**에서 사용자와 타깃, 실제 판매 SKU, 핵심 문제·소구, 판매가·묶음·판매 옵션, 포지셔닝, 브랜드 방향과 1차 실험 범위를 결정한다. 이 단계는 사용자와 정하는 오퍼 결정 게이트다.
- 검색 전에는 오퍼 결정을 필수 입력으로 요구하지 않는다. 게이트 결정 전 상태는 `AWAITING_USER_OFFER_DECISION`이며 `product-plan.json`을 승인 가능 상태로 승격하지 않는다.
- 결정 결과를 반영한 전체 `product-plan.json`을 보여준 뒤 현재 SHA-256에 대한 별도 사용자 승인을 받는다. 조사 결과 선택과 최종 계획 승인을 한 동작으로 합치지 않는다.

## 주장-근거 매트릭스

각 행에 다음을 기록한다.

| 필드 | 의미 |
|---|---|
| `pain_id` | 경쟁 저별점 반복 불만 |
| `claim_id` | 우리 소구 후보 |
| `sku_resolution` | 가능·조건부·불가·미확인 |
| `evidence_ids` | 검사서·시험·이미지·사용자 확인 |
| `phase_1_test` | 저비용 간이 실험 |
| `phase_2_test` | 판매 신호 후 심화 검증 |
| `publish_state` | 게시 가능·조건부·금지 |
| `rollback` | 실패 시 돌아갈 단계 |

경쟁 불만 빈도는 관찰 표본 안에서만 계산한다. 한두 리뷰를 시장 전체 사실로 일반화하지 않는다.
