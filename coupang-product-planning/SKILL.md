# 쿠팡 제품기획

경쟁사 불만을 그대로 카피하지 말고 `반복 불만 → 구매 기준 → 우리 SKU가 해결 가능한가 → 근거 → 제한된 주장`으로 변환한다.

## 최소 시작점과 오퍼 결정

- 공급처 상세 URL 하나만으로 조사를 시작할 수 있다. 상세 본문·옵션·상품정보·공개 후기·문의를 읽어 실제 상품의 색상·사이즈·구성·소재·관리법과 가능한 소구점을 정리한다.
- 조사 결과를 사용자에게 먼저 보고 소구점 후보를 제안한다. 사용자가 판매가·묶음·판매 옵션과 소구점을 선택하기 전에는 `오퍼 결정 게이트`를 통과시키거나 최종 상품기획을 확정하지 않는다.

## 작업 순서

1. 공급처 상세 URL 하나만 받은 경우 위 최소 조사를 먼저 수행한다. 승인된 소싱 후보·가격안이 있으면 함께 잠근다.
2. Browser Harness로 공개된 경쟁상품의 별점 1~3점 리뷰를 직접 조사하고 URL·조사시각·표본 범위를 보존한 뒤 문제 유형·빈도·반품 위험으로 묶는다. 로그인·CAPTCHA·접근 통제는 우회하지 않는다.
3. 각 문제를 실제 SKU가 해결할 수 있는지 `가능 / 조건부 / 불가 / 미확인`으로 판정한다.
4. `project.json`의 `folderMap.sourceAssets`에 드래그앤드롭된 실제 SKU·라벨 이미지를 미리 확인하고, 가능한 항목만 소구 후보로 만들어 검사서·시험성적서·간이 실험·실측·라벨 근거를 연결한다.
5. 타깃, 대표 문제, 포지셔닝, 오퍼, 옵션, 사이즈, 관리법, 게시 가능 주장과 금지 주장을 `product-plan.json`에 기록한다.
6. 사용자에게 계획 전체를 한 번에 보여주고 현재 파일 해시에 대한 명시적 승인을 받는다.

## 입력 경계

- 공급처 상세 URL 하나만으로 조사를 시작할 수 있다. Codex는 상세 본문·옵션·상품정보·공개 후기·문의를 확인해 색상·사이즈·구성·소재·관리법을 정리하고, 조사 결과를 사용자에게 먼저 보고한다. 그다음 판매가·묶음·판매 옵션을 사용자와 정하는 오퍼 결정 게이트를 통과한다.
- 사용자에게 리뷰 URL이나 조사 파일 경로를 입력받지 않는다. 경쟁사 저평점 리뷰 수집과 근거 정리는 Codex의 조사 작업이다.
- 사용자에게 이미지 폴더 경로를 입력받지 않는다. UI가 저장한 `folderMap.sourceAssets`의 기존 파일을 읽고, 부족하면 필요한 촬영 구도·라벨 면을 구체적으로 요청한다.
- 사용자가 사이즈·구성·소재·관리법을 각각 제공했는지 확인한다. 값마다 실측·라벨·공급처 원문 중 하나 이상의 Evidence ID를 연결하지 못하면 확정 사실로 승격하지 않는다.

## 1차와 2차

- `phase_1`: 최소 SKU 사실과 가능한 간이 실험만 수행한다. 실험 불가 소구는 미검증 가설로 남기고 판매 주장에는 쓰지 않는다.
- `phase_2`: 사용자가 제공한 캠페인별 14일 내 판매 2개 이상과 ROAS 400% 이상을 모두 충족한 경우에만 진입한다. 자사 후기, 다수 샘플, 검사서 적용 범위, 심화 실험, 실제 사이즈·관리법을 다시 검증한다.
- 필수 안전·법정 표시·제품 동일성은 2차로 미루지 않는다.

## Acceptance Criteria

- 핵심 소구마다 경쟁 불만·SKU 해결 가능성·Evidence ID 연결: 100%
- `불가/미확인` 소구의 판매 카피 사용: 0건
- 수치·성능 주장마다 적용 SKU가 확인된 근거: 100%
- 사이즈 값마다 단위·측정 시작점·끝점·허용 오차 상태 기록: 100%
- 관리법마다 케어라벨·공급처·실험 중 하나의 출처 연결: 100%
- 사용자 승인이 현재 `product-plan.json` SHA-256에 결속됨: 필수

상세 산출물과 롤백은 [references/product-planning-contract.md](references/product-planning-contract.md)를 따른다.
